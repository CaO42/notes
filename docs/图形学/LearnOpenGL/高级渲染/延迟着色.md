### 延迟着色

我们此前一直使用的光照方式叫做**正向渲染(Forward Rendering)**或者**正向着色法(Forward Shading)**，它是一种非常直接的方式，在场景中我们根据所有光源渲染一个物体，之后再渲染下一个物体，以此类推。

这种方法便于理解，但是很少被现代实时渲染采用。这是由于在场景中具有大量物体和大量光源的时候，计算量会快速增加。

实际上，我们最终得到的片段数量永远只和屏幕的大小相关。可我们进行完整光照计算（片段着色器）的次数却可能远多于这个数量。

**延迟着色法(Deferred Shading)**，**或者说是延迟渲染(Deferred Rendering)**，为了解决上述问题而诞生了。

延迟着色法基于我们**延迟(Defer)**或**推迟(Postpone)**大部分计算量非常大的渲染(像是光照)到后期进行处理的想法。

它将渲染分为两个处理阶段(Pass)：在第一个**几何处理阶段(Geometry Pass)**中，我们先渲染场景一次，之后获取对象的各种几何信息，并储存在一系列叫做**G缓冲(G-buffer)**的纹理中；比如位置向量(Position Vector)、颜色向量(Color Vector)、法向量(Normal Vector)和/或镜面值(Specular Value)。

场景中这些储存在G缓冲中的几何信息将会在之后用来完成光照计算。

我们会在第二个**光照处理阶段(Lighting Pass)**中使用G缓冲内的纹理数据。在光照处理阶段中，我们渲染一个屏幕大小的方形，并使用G缓冲中的几何数据对每一个片段计算场景的光照；在每个像素中我们都会对G缓冲进行迭代。光照计算过程和以前一样，但是现在我们需要从对应的G缓冲而不是顶点着色器(和一些uniform变量)那里获取输入变量了。

为什么这可以减少计算量？这是因为我们在填充G缓冲的时候进行深度测试，最后的G缓冲只有应该显示在最前面片段。这样保证了在光照处理阶段中，每一个像素都只处理一次，能够省下大量的光照计算。

在几何处理阶段中填充G缓冲非常高效，因为我们只需要直接储存像素位置，颜色或者是法线等对象信息到帧缓冲中，而这几乎不会消耗处理时间。在此基础上使用多渲染目标(Multiple Render Targets, MRT)技术，我们甚至可以在一个渲染处理之内完成这所有的工作。

> 我们可能会想到此前学过的**提前深度测试(Early-Z)**，看起来它也是一种能够节省光照计算消耗的硬件特性。它确实一定程度上可以，但也有很多问题，不能替代延迟着色。
>
> 深度写入限制：如果片段着色器修改了深度值（如写入 `gl_FragDepth`），GPU 将无法提前预知深度，从而强制禁用 Early-Z。
>
> 几何与光照未解耦：在传统前向渲染中，为了让 Early-Z 发挥最大效用以避免过度绘制 (Overdraw)，必须在 CPU 端对物体进行从前向后的排序，这极其消耗性能且无法处理穿插遮挡。
>
> 替代方案与局限：我们可以通过 Z-Prepass（深度预通道） 渲染两次几何体（一次只存入深度，一次计算颜色并开启提前深度测试）来解决排序和过度绘制问题，但这增加了顶点处理的开销。（即使第二次会有大量片段被抛弃，我们依然跑了两遍全场景的顶点着色器，和一遍片段着色器）更重要的是，它依然无法像延迟着色那样，把光照计算的复杂度从基于‘物体’转移到纯‘屏幕空间’，因此在处理大量光源时依然力不从心。

整个渲染过程大致如下：

```c++
while(...) // 游戏循环
{
    // 1. 几何处理阶段：渲染所有的几何/颜色数据到G缓冲 
    glBindFramebuffer(GL_FRAMEBUFFER, gBuffer);
    glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);
    gBufferShader.Use();
    for(Object obj : Objects)
    {
        ConfigureShaderTransformsAndUniforms();
        obj.Draw();
    }  
    // 2. 光照处理阶段：使用G缓冲计算场景的光照
    glBindFramebuffer(GL_FRAMEBUFFER, 0);
    glClear(GL_COLOR_BUFFER_BIT);
    lightingPassShader.Use();
    BindAllGBufferTextures();
    SetLightingUniforms();
    RenderQuad();
}
```

#### 缺陷

我们知道，所有的缓冲在Draw Call的调用中都会被打包传输进显存。由于G缓冲要求我们在纹理颜色缓冲中存储相对比较大的场景数据，这会**消耗比较多的显存**，尤其是类似位置向量之类的需要高精度的场景数据。

> 实际上，延迟着色的物理瓶颈往往不是显存**容量**不够，而是**显存带宽（Memory Bandwidth）**。
>
> 在几何阶段，GPU 需要同时向 3~4 张高精度的 RGBA 纹理（位置、法线、颜色、材质参数）写入数据；在光照阶段又要同时读取它们。这种极高频的读写会导致总线变堵，这就是图形学中常说的 **Bandwidth Bound（带宽受限）**。

延迟渲染也**不支持混合**操作（至少我们目前采用的算法不支持），导致我们无法渲染半透明的物体。

也**无法使用MSAA**，因为我们在光照处理阶段已经失去了原有的几何边缘信息，无法计算子采样点的覆盖率。

这些缺陷需要额外的算法进行解决，我们之后会进行讨论。

#### G缓冲

G缓冲需要存储我们会在光照计算中使用的所有数据。在Blinn-Phong Lighting的渲染模型中，我们需要如下的数据：

- 一个3D**位置**向量来计算(插值)片段位置变量供`lightDir`和`viewDir`使用
- 一个RGB漫反射**颜色**向量，也就是反照率(Albedo)
- 一个3D**法**向量来判断平面的斜率
- 一个镜面强度(Specular Intensity)浮点值
- 所有光源的位置和颜色向量
- 玩家或者观察者的位置向量

其中，光源的位置，颜色，和玩家的观察位置对每个片都相同，可以通过uniform变量来设置。

在OpenGL（或者说图形学）中，纹理其实并没有限制我们存储的内容究竟代表着什么含义。纹理并不是仅用于存储颜色值的二维数组。实际上它存储n维的向量（通常为3或4维），只是颜色变量刚好符合条件而已。我们此前的法线贴图、视差贴图就是最好的例子。

```c++
// 为了实现多步的处理阶段 我们需要离屏渲染
// 我们创建一个帧缓冲 然后为其绑定对应的颜色插件（虽然其存储的并不是颜色）
GLuint gBuffer;
glGenFramebuffers(1, &gBuffer);
glBindFramebuffer(GL_FRAMEBUFFER, gBuffer);
GLuint gPosition, gNormal, gColorSpec;

// - 位置颜色缓冲
glGenTextures(1, &gPosition);
glBindTexture(GL_TEXTURE_2D, gPosition);
// 此处我们纹理使用的内部数据格式为GL_RGB16F 它代表每个分量采用f16相当于float进行存储
// 常规的GL_RGB只有8位 由于位置变量需要更高的精度 所以我们使用GL_RGB16F格式
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGB16F, SCR_WIDTH, SCR_HEIGHT, 0, GL_RGB, GL_FLOAT, NULL);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_NEAREST);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_NEAREST);
// 绑定至0号颜色插件 这里绑定的位置与之后片段着色器中需要对应
glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT0, GL_TEXTURE_2D, gPosition, 0);

// - 法线颜色缓冲
glGenTextures(1, &gNormal);
glBindTexture(GL_TEXTURE_2D, gNormal);
// 同理使用GL_RGB16F                       
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGB16F, SCR_WIDTH, SCR_HEIGHT, 0, GL_RGB, GL_FLOAT, NULL);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_NEAREST);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_NEAREST);
glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT1, GL_TEXTURE_2D, gNormal, 0);

// - 颜色 + 镜面颜色缓冲
glGenTextures(1, &gAlbedoSpec);
glBindTexture(GL_TEXTURE_2D, gAlbedoSpec);
// 这里我们采用GL_RGBA格式 将镜面反射强度存储在A分量中
// 实际上 如果我们采用了镜面反射贴图 还需要创建一个额外的纹理用于存储
// 后续G缓冲中增加更多数据时 我们也可以这样组合来减少纹理数量
glTexImage2D(GL_TEXTURE_2D, 0, GL_RGBA, SCR_WIDTH, SCR_HEIGHT, 0, GL_RGBA, GL_FLOAT, NULL);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MIN_FILTER, GL_NEAREST);
glTexParameteri(GL_TEXTURE_2D, GL_TEXTURE_MAG_FILTER, GL_NEAREST);
glFramebufferTexture2D(GL_FRAMEBUFFER, GL_COLOR_ATTACHMENT2, GL_TEXTURE_2D, gAlbedoSpec, 0);

// - 告诉OpenGL我们将要使用(帧缓冲的)哪种颜色附件来进行渲染
// 这是使用MRT的必须操作 必须与片段着色器中布局相符
GLuint attachments[3] = { GL_COLOR_ATTACHMENT0, GL_COLOR_ATTACHMENT1, GL_COLOR_ATTACHMENT2 };
glDrawBuffers(3, attachments);

// 之后同样添加渲染缓冲对象(Render Buffer Object)为深度缓冲(Depth Buffer)，并检查完整性
[...]
```

这是一个简易的Blinn-Phong Lighting，可能没有我们之前学习的完整，但是已经足够展示延迟渲染的实现了。

#### 几何处理阶段

这个阶段比较简单，只需要存储数据到G缓冲中即可。

```c++
#version 330 core
// 这里使用布局指示符(Layout Specifier) 来指示输出布局是MRT的需要
// 这里的布局需要与绑定的attachments数组内的顺序对应
layout (location = 0) out vec3 gPosition;
layout (location = 1) out vec3 gNormal;
layout (location = 2) out vec4 gAlbedoSpec;

in vec2 TexCoords;
in vec3 FragPos;
in vec3 Normal;

uniform sampler2D texture_diffuse1;
uniform sampler2D texture_specular1;

void main()
{    
    // 存储第一个G缓冲纹理中的片段位置向量
    gPosition = FragPos;
    // 同样存储对每个逐片段法线到G缓冲中
    gNormal = normalize(Normal);
    // 和漫反射对每个逐片段颜色
    gAlbedoSpec.rgb = texture(texture_diffuse1, TexCoords).rgb;
    // 存储镜面强度到gAlbedoSpec的alpha分量
    gAlbedoSpec.a = texture(texture_specular1, TexCoords).r;
}

// 之后对每一个物体正常进行渲染，但是使用G缓冲的帧缓冲，以及这个片段着色器
```

因为有光照计算，所以保证**所有变量在一个坐标空间**当中至关重要。在这里我们在世界空间中存储(并计算)所有的变量。

我们可能想采用之前学习法线贴图时的切线空间进行计算，但实际上每一个平面的切线空间不同，我们无法共用相同的光源位置。为了解决这个问题，使用切线空间时我们可能需要向G缓冲存储额外的TBN矩阵等数据。

#### 光照处理阶段

现在我们已经得到了G缓冲中的一大堆数据了。由于几何处理阶段填充时开启了深度测试，现在处于G缓冲中的就是我们需要的最终片段的数据了。我们接下来只用对每个最后呈现在屏幕上的片段进行昂贵的光照计算就行了。

```c++
// 几何处理结束后记得回到默认帧缓冲
glBindFramebuffer(GL_FRAMEBUFFER, 0);

glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);
shaderLightingPass.Use();
glActiveTexture(GL_TEXTURE0);
glBindTexture(GL_TEXTURE_2D, gPosition);
glActiveTexture(GL_TEXTURE1);
glBindTexture(GL_TEXTURE_2D, gNormal);
glActiveTexture(GL_TEXTURE2);
glBindTexture(GL_TEXTURE_2D, gAlbedoSpec);
// 同样发送光照相关的uniform 将对应的sample设置为对应纹理
SendAllLightUniformsToShader(shaderLightingPass);
glUniform3fv(glGetUniformLocation(shaderLightingPass.Program, "viewPos"), 1, &camera.Position[0]);
// 渲染屏幕四边形，需要应用接下来的光照计算片段着色器
RenderQuad();  
```

```C++
#version 330 core
out vec4 FragColor;
in vec2 TexCoords;

uniform sampler2D gPosition;
uniform sampler2D gNormal;
uniform sampler2D gAlbedoSpec;

struct Light {
    vec3 Position;
    vec3 Color;
};
const int NR_LIGHTS = 32;
uniform Light lights[NR_LIGHTS];
uniform vec3 viewPos;

void main()
{             
    // 从G缓冲中获取数据
    vec3 FragPos = texture(gPosition, TexCoords).rgb;
    vec3 Normal = texture(gNormal, TexCoords).rgb;
    vec3 Albedo = texture(gAlbedoSpec, TexCoords).rgb;
    float Specular = texture(gAlbedoSpec, TexCoords).a;

    // 然后和往常一样地计算光照
    vec3 lighting = Albedo * 0.1; // 硬编码环境光照分量 实际上这里相当于多计算了一个环境光光源ß
    vec3 viewDir = normalize(viewPos - FragPos);
    for(int i = 0; i < NR_LIGHTS; ++i)
    {
        // 漫反射
        vec3 lightDir = normalize(lights[i].Position - FragPos);
        vec3 diffuse = max(dot(Normal, lightDir), 0.0) * Albedo * lights[i].Color;
        // 镜面反射
        vec3 halfwayDir = normalize(lightDir + viewDir);  
        float spec = pow(max(dot(Normal, halfwayDir), 0.0), 16.0);
        vec3 specular = lights[i].Color * spec * Specular;
        // 衰减
        float distance = length(lights[i].Position - FragPos);
        float attenuation = 1.0 / (1.0 + lights[i].Linear * distance + lights[i].Quadratic * distance * distance);
        diffuse *= attenuation;
        specular *= attenuation;
        lighting += diffuse + specular;
    }

    FragColor = vec4(lighting, 1.0);
}  
```

#### 结合延迟和前向渲染

延迟渲染和前向渲染实际上都是为了写入最后的屏幕缓冲区，也可以结合在一起使用。

比如我们现在希望在延迟渲染所采用的所有点光源处，都渲染一个纯色的立方体。我们当然可以在此前的延迟渲染时就加入这些立方体，或者再进行一次延迟渲染的过程。我们也可以在我们完成延迟渲染操作之后，用前向渲染完成这个简单的过程。

```c++
// 延迟渲染光照渲染阶段
[...]
RenderQuad();

// 这一步是关键 它把G缓冲中的深度插件给写入默认深度帧缓冲中 使得我们前向渲染能够正常进行深度测试
glBindFramebuffer(GL_READ_FRAMEBUFFER, gBuffer);
glBindFramebuffer(GL_DRAW_FRAMEBUFFER, 0); // 写入到默认帧缓冲
glBlitFramebuffer(
  0, 0, SCR_WIDTH, SCR_HEIGHT, 0, 0, SCR_WIDTH, SCR_HEIGHT, GL_DEPTH_BUFFER_BIT, GL_NEAREST
);
glBindFramebuffer(GL_FRAMEBUFFER, 0);

// 现在像正常情况一样正向渲染所有光立方体
shaderLightBox.Use();
glUniformMatrix4fv(locProjection, 1, GL_FALSE, glm::value_ptr(projection));
glUniformMatrix4fv(locView, 1, GL_FALSE, glm::value_ptr(view));
for (GLuint i = 0; i < lightPositions.size(); i++)
{
    model = glm::mat4();
    model = glm::translate(model, lightPositions[i]);
    model = glm::scale(model, glm::vec3(0.25f));
    glUniformMatrix4fv(locModel, 1, GL_FALSE, glm::value_ptr(model));
    glUniform3fv(locLightcolor, 1, &lightColors[i][0]);
    RenderCube();
}
```

#### 光体积

这是延迟渲染对比普通前向渲染的真正优势所在。

为什么我们当场景中数千个光源时，采用原本的光照计算方式进行实时渲染不现实？这是因为即使我们将需要渲染的最终片段数量降低到了屏幕大小，每多一个光源时，我们依然要对每个片段计算该光源的光照。在光源数量达到上千个时，这个计算量依然庞大。

**光体积(Light Volumes)**就是用来解决这个问题的技术。

此前，当我们渲染一个复杂光照场景下的片段着色器时，我们会计算场景中**每一个**光源的贡献，不管它们离这个片段有多远。但由于光的衰减，很大一部分的光源根本就不会到达这个片段（对该片段最终颜色的贡献极小），我们浪费了大量的光照运算。

##### 光体积半径

我们可以这样考虑，对于空间中每个点光源，我们假设一个光体积的半径。在这个半径的球体内的片段，我们才认为它被这个光源照射到，才进行对应的光照计算。现在的关键在于如何决定光体积半径。

根据我们之前的光照衰减公式：$F_{light} = \frac{I}{K_c + K_l * d + K_q * d^2}$ 只有$d$趋向无穷大时$F_{light}$才会趋向0，我们需要假定一个阈值，低于这个亮度时我们认为光源对更远的变量不再产生影响。

我们假定，$\frac{5}{256} = \frac{I_{max}}{Attenuation} 即F_{light}= \frac{5}{256}$为最远端的光照强度。

解这个方程得$x = \frac{-K_l + \sqrt{K_l^2 - 4 * K_q * (K_c - I_{max} * \frac{256}{5})}}{2 * K_q}$，我们可以根据它算出每个光源的光体积半径。

```c++
float constant  = 1.0; 
float linear    = 0.7;
float quadratic = 1.8;
float lightMax  = std::fmaxf(std::fmaxf(lightColor.r, lightColor.g), lightColor.b);
float radius    = 
  (-linear +  std::sqrtf(linear * linear - 4 * quadratic * (constant - (256.0 / 5.0) * lightMax))) 
  / (2 * quadratic);  

// 这是在cpu中完成的预处理过程 在渲染过程中，我们已经提前知道了每一个光源的半径
```

我们可能会以为可以如下的实现光体积：

```c++
// frag
struct Light {
    [...]
    float Radius;
}; 

void main()
{
    [...]
    for(int i = 0; i < NR_LIGHTS; ++i)
    {
        // 计算光源和该片段间距离
        float distance = length(lights[i].Position - FragPos);
        if(distance < lights[i].Radius)
        {
            // 执行大开销光照
            [...]
        }
    }   
}
```

但这种实现并不可行。

片段着色器运行在GPU上，但是GPU高度并行不擅长进行分支操作。如果采用这种大量使用if的方式进行计算，效率可能比直接计算每个光源更低。在现代的GPU中，实际上对一致分支/收益大的分支可以处理得很好，可以避免这个问题。但全屏 quad 中即使加 if(distance < radius)，仍然要对“每个屏幕像素 × 每个光源”执行循环和距离判断，复杂度没有被光体积裁剪掉。

我们采用渲染一个**光体积球体**的方式解决这个问题。

##### 光体积球体

在几何处理阶段之后，我们现在不进行屏幕四边形的渲染，也不在屏幕四边形的着色器中计算光照。

我们为每一个光源渲染一个球体，并且根据其半径缩放到与光体积半径同样的大小。然后将它渲染到屏幕上，对于被渲染的每一个片段，我们从它对应的G缓冲中取值，采用和我们此前一样的光照计算。只有在光体积屏幕投影内的像素会执行，仍需用 G-buffer 位置或模板技术进一步剔除

```c++
// 绑定默认帧缓冲 
glBindFramebuffer(GL_FRAMEBUFFER, 0);
glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);

// 开启混合，并设置为累加模式
// 这是由于一个片段可能不只被一个光源照射 必须开启混合
glEnable(GL_BLEND);
glBlendFunc(GL_ONE, GL_ONE); 

// 我们不需要写入深度缓冲了，因为球体只是用来计算光照的“范围”
glDepthMask(GL_FALSE); 
// 禁用深度测试（或者设置特殊的背面剔除，见后文的问题解决）
glDisable(GL_DEPTH_TEST);
```

```c++
lightVolumeShader.use(); // 绑定光体积专用的着色器

// 绑定 G-Buffer 中的纹理供着色器读取
glActiveTexture(GL_TEXTURE0);
glBindTexture(GL_TEXTURE_2D, gPosition);
// ... 绑定 gNormal, gAlbedo 等 ...

for(unsigned int i = 0; i < lights.size(); i++) {
    // 1. 计算当前光体积（球体）的变换矩阵
    glm::mat4 model = glm::mat4(1.0f);
    // 移动到光源中心点
    model = glm::translate(model, lights[i].Position); 
    // 根据光源的衰减半径放大球体
    model = glm::scale(model, glm::vec3(lights[i].Radius)); 
    
    // 2. 传给着色器
    lightVolumeShader.setMat4("model", model);
    lightVolumeShader.setVec3("light.Position", lights[i].Position);
    lightVolumeShader.setVec3("light.Color", lights[i].Color);
    lightVolumeShader.setFloat("light.Radius", lights[i].Radius);

    // 3. 渲染球体模型！
    // GPU 会把这个球体光栅化，只有在球体范围内的像素才会执行下面的片段着色器
    RenderSphere(); 
}

// 渲染完后恢复状态
glEnable(GL_DEPTH_TEST);
glDepthMask(GL_TRUE);
glDisable(GL_BLEND);
```

// frag
out vec4 FragColor;

uniform sampler2D gPosition;
uniform sampler2D gNormal;
uniform sampler2D gAlbedoSpec;

uniform vec3 lightPosition;
uniform vec3 lightColor;
uniform float lightRadius;

void main() {
    // 因为我们不再采用屏幕矩形，我们需要通过 gl_FragCoord 手动计算当前像素对应的纹理坐标
    vec2 texCoords = gl_FragCoord.xy / textureSize(gPosition, 0);

    // 从 G-Buffer 获取场景信息
    vec3 FragPos = texture(gPosition, texCoords).rgb;
    vec3 Normal = texture(gNormal, texCoords).rgb;
    vec3 Albedo = texture(gAlbedoSpec, texCoords).rgb;

    // 计算光照（只计算当前这一个光）
    float distance = length(lightPosition - FragPos);
    if(distance > lightRadius) {
        discard; // 超出范围直接丢弃
      	// 我们需要注意的是，球体只是被投影到了屏幕平面上
      	// 并不代表着在这个投影到圆柱中的片段处于光体积球体之中
    }
    
    // ... 具体的漫反射、高光计算 ...
    vec3 lighting = Albedo * lightColor * attenuation; 
    
    // 输出的颜色会自动和屏幕上已有的颜色相加（因为之前设置了 glBlendFunc(GL_ONE, GL_ONE)）
    FragColor = vec4(lighting, 1.0);
}

##### 面剔除

由于我们关闭了深度值写入和深度测试，在不开启面剔除的情况下，我们对球体的正面和背面都会进行渲染。这会导致每个片段被计算两次光照并叠加进缓冲区，产生错误的结果。

我们可能会像通常一样开启背面剔除。但是这样会产生一个问题，当我们在球体之外时，渲染正常进行。但是当我们位于球体内部时，球体正面被视锥体剔除了，球体背面也被剔除了。此时这个光源完全失去了对片段的影响，产生错误结果。

我们可以采用开启正面剔除的方式简单处理这个问题。

```c++
glEnable(GL_CULL_FACE)
glCullFace(GL_FRONT);  // 只渲染球的内侧（背面）
glDisable(GL_DEPTH_TEST); 
RenderSphere();
glCullFace(GL_BACK);   // 恢复默认
```

这样无论你站在球外面还是球里面，总能渲染出球的背面，片段着色器就能正常触发。

实际上还有一种模板缓冲技巧（Stencil Buffer）是一种更为高级且性能更好的解决方案，它能精确限制光体积只在相交的几何体上渲染，但我们目前采用这种简单的处理就可以了。

> 这种技巧原理是：先利用深度测试将光体积球体的背面渲染进模板缓冲，然后再渲染正面并利用模板缓冲进行剔除，这能完美解决摄像机位置带来的面剔除困境，被称为 **Stencil Light Pre-pass**。

##### 不同类型光源在光体积延迟渲染中的处理

光体积并不是所有光源都需要使用的技术。它的意义在于利用光源的有限影响范围，减少不必要的片段光照计算。因此，不同类型光源通常采用不同的处理方式。

- **环境光 / 间接光近似**
  环境光不来自某个具体位置，也没有明确的影响半径，通常应单独做一次全屏处理。
  例如先渲染一个屏幕四边形，从 G-buffer 读取 Albedo，并写入 `Albedo * ambientStrength`。之后再开启加法混合，累加局部光源的贡献。

- **方向光 Directional Light**
  方向光可以认为来自无限远处，光照方向在整个场景中一致，也没有有限的光体积半径。因此方向光通常也用全屏 pass 计算。
  片段着色器从 G-buffer 读取位置、法线、材质参数，然后用统一的 `lightDir` 对每个屏幕像素计算一次方向光贡献。

- **点光源 Point Light**
  点光源具有位置，并且随着距离衰减。我们可以根据衰减阈值计算出一个有限半径，然后渲染对应的球形光体积。只有落在球体屏幕投影范围内的像素才会执行该光源的片段着色器，片段着色器中再用 G-buffer 位置判断真实距离是否小于半径。

- **聚光灯 Spot Light**
  聚光灯也属于局部光源，但它的影响范围不是球体，而是一个锥体或截锥体。因此可以渲染一个与聚光灯角度和距离匹配的锥形光体积。
  在片段着色器中仍然需要从 G-buffer 读取世界空间位置，判断该点是否位于聚光灯方向和 cutoff 角度范围内，并计算距离衰减与角度衰减。
