在之前，我们以向三角形上附加**纹理贴图**的方式来增加额外细节，提升真实感。

但无论是漫反射贴图，镜面光贴图，放射光贴图，都只能改变平面上的颜色。只要我们靠近物体就能感受到平坦的表面带来的不真实感。现实中的物体具有凹凸不平的表面，为了更好的显示效果，我们需要的是某种可以告知光照系统给所有有关**物体表面**类似**深度**这样的细节的方式。

### 法线贴图

如果我们以光的视角来看这个问题：是什么使表面被视为完全平坦的表面来照亮？答案会是表面的**法线向量**。平面只有一个法线向量，表面完全根据这个法线向量被以一致的方式照亮。如果每个fragment都是用自己的不同的法线会怎样？这样我们就可以根据表面细微的细节对法线向量进行改变；这样就会获得一种表面看起来要复杂得多的幻觉：

![img](https://learnopengl-cn.github.io/img/05/04/normal_mapping_surfaces.png)

当每个fragment使用了自己的法线时，计算出的光照效果相当于一个表面由很多微小的（垂直于法线向量的）平面所组成的物体，物体表面的细节将会得到极大提升。

这种为每个片段采用不同法向量，代替整个平面使用同一个法线，来模拟表面凹凸感的技术叫做**法线贴图（normal mapping）**或凹凸贴图（bump mapping）。

这种方法能使细节获得极大提升，开销却不大。因为我们只需要改变每个fragment的法线向量。

#### 法线贴图

我们先考虑直接用一张普通的贴图存储一个固定平面对应的所有法线（在世界空间中的法线）。由于纹理本质存储的是rgb值范围与法线不同，法线向量的范围在-1到1之间，所以我们先要将其映射到0到1的范围。

```c++
vec3 rgb_normal = normal * 0.5 + 0.5; // 从 [-1,1] 转换至 [0,1]
```

在片段着色器中使用时，我们需要将其复原使用。

```c++
// 从法线贴图范围[0,1]获取法线
normal = texture(normalMap, fs_in.TexCoords).rgb;
// 将法线向量转换为范围[-1,1]
normal = normalize(normal * 2.0 - 1.0);  

// ... 作为法向量进行光照运算
```

如果我们应用这种法线贴图到那个平面上，可以看到不错的效果。

然而有个问题限制了这种法线贴图的使用，我们改变平面的方向，比如把这个贴图应用在一个朝向不同的平面上。由于我们记录的是绝对的世界空间法线，此时的光线计算就会产生问题。

我们可以联想到之前的求法向量的方法：假设我们有一个平面，以及平面的一个世界空间法线贴图。我们假设目前的面是默认的面，而另一个面（比如朝向不同的平面）是由这个平面变换得到的。我们可以让另一个平面上每一个片段采样得到的法向量的值，都乘上model矩阵的逆转置矩阵，这样我们就可以得到正确的法向量。

但是，一个几何图形，不止有一个默认位置的面。采用这种做法时，即使我们希望每一个面的纹理相同，但是我们仍然需要为每个面初始化一张法线贴图（因为朝向导致世界空间的默认法线不同）。这导致这种解决方式极度浪费资源和麻烦。

我们可以采用更优秀的解决方案。

#### 切线空间法线贴图（Tangent Space Normal Mapping）

对于我们渲染的三角形，先假设它没有被拉伸，只是进行了旋转位移等操作。

我们可以想象三角形实际上是法线贴图的一部分，我们在变换的实际上是一个正方形的面。

无论如何变换，相对于平面的空间内的东西是不变的。我们可以将法线贴图的值定义为该空间的法向量。

![img](https://learnopengl-cn.github.io/img/05/04/normal_mapping_tbn_vectors.png)

假设我们知道当前三角形所在平面的TBN值，我们就可以将法线贴图正确变换到正常的世界空间里进行计算。

#### TBN矩阵

N向量我们之前已经求过，将模型空间的法向量乘上模型矩阵的逆转置矩阵即可得到世界空间下正确的平面法向量。

关于TB向量：

![img](https://learnopengl-cn.github.io/img/05/04/normal_mapping_surface_edges.png)

如图，绿色为B向量（或与B平行），红色为T向量（或与T平行），我们得到如下两个方程。

$E_1 = \Delta U_1T + \Delta V_1B$ 		$E_2 = \Delta U_2T + \Delta V_2B$

两个方程两个变量，我们可以求出T和B的值。

方程组转换为矩阵形式可以计算出如下结果：

$\begin{bmatrix} T_x & T_y & T_z \\ B_x & B_y & B_z \end{bmatrix}  = \frac{1}{\Delta U_1 \Delta V_2 - \Delta U_2 \Delta V_1} \begin{bmatrix} \Delta V_2 & -\Delta V_1 \\ -\Delta U_2 & \Delta U_1 \end{bmatrix} \begin{bmatrix} E_{1x} & E_{1y} & E_{1z} \\ E_{2x} & E_{2y} & E_{2z} \end{bmatrix}$

$令 f = \frac{1}{\Delta U_1 \Delta V_2 - \Delta U_2 \Delta V_1}$

$得 T_x = f(\Delta V_2E_{1x}-\Delta V_1E_{2x})$ ……

> 我们可能会想到，TBN矩阵的计算需要多个顶点的数据，所以应该在几何着色器中进行计算。
>
> 这是可行的，但是并不是最佳的实现方案。
>
> 实际上我们在CPU处理阶段（Draw Call之前），就可以计算出模型空间中的TBN向量。
>
> 而根据法向量变换的原理，我们分别为它们乘上模型矩阵就可以得到所需的TBN向量。（此处假设不存在非等比缩放）
>
> 这种做法可以大幅减少计算量，并且避免强制使用几何着色器的性能问题。

#### 一点优化

现在我们得到了TBN矩阵，我们可以直接使用它来乘上我们采样得到的法线来得到世界空间下的法向量。

```c++
// vert
out mat3 TBN;

void main(){
  	TBN = mat3(T, B, N);
}

//frag
in mat3 TBN;
normal = texture(normalMap, fs_in.TexCoords).rgb;
normal = normalize(normal * 2.0 - 1.0);   
normal = normalize(TBN * normal);
```

但是这实际上并不是最高效的使用方法。我们甚至可以不用给片段着色器传入TBN矩阵。

目前，我们片段着色器在世界空间下计算光照，所以需要传入TBN矩阵来对采样的法向量进行变换。

我们可以直接在顶点着色器中，将传入片段着色器的向量都变换为当前的切线空间向量。

```c++
// vert
out VS_OUT {
    vec3 FragPos;
    vec2 TexCoords;
    vec3 TangentLightPos;
    vec3 TangentViewPos;
    vec3 TangentFragPos;
} vs_out;

uniform vec3 lightPos;
uniform vec3 viewPos;

[...]

void main()
{    
    [...]
    mat3 TBN = transpose(mat3(T, B, N)); // 此处为逆矩阵（TBN正交 逆矩阵=转置矩阵） 用于转换进切线空间
    vs_out.TangentLightPos = TBN * lightPos;
    vs_out.TangentViewPos  = TBN * viewPos;
    vs_out.TangentFragPos  = TBN * vec3(model * vec4(position, 0.0));
}
```

这看起来，将一次TBN的变换转换为了三次（或更多次）的变换。但实际上，顶点着色器的out变量会自动插值产生片段中对应的in变量，重心插值的速度要快于每一个片段都执行TBN变换。

#### 剩余的问题

##### 顶点共享与平滑化

为了节省内存，现代渲染通常使用索引缓冲（Index Buffer），一个顶点会被多个相邻的三角形共享。

这个顶点的顶点属性不可能同时存储所有相邻三角形的TBN（顶点属性大小通常固定）。实际上，它存储的是这几个面的TBN值平均化后的结果。

此时的T、B、N之间极大可能不再相互垂直。TBN不再是正交矩阵。TBN的转置也不再为TBN的逆矩阵。

但是切线空间的三个基向量是正交的，我们仍然需要一个正交的TBN矩阵。

我们可以采用一种叫做格拉姆施密特正交化的方法：

```c++
vec3 T = normalize(vec3(model * vec4(aTangent, 0.0)));
vec3 N = normalize(vec3(model * vec4(aNormal, 0.0)));

// Gram-Schmidt 正交化，确保 T 垂直于 N
T = normalize(T - dot(T, N) * N);

// 通过叉乘获取完全正交的 B 向量（有时需要乘上一个手性系数，手性系数为1.0/-1.0）
// vec3 B = cross(N, T) * aTangent.w; 手性系数存储在w分量中
// 当使用镜像模型时，为了节省模型空间，对两个互为镜像的面使用同一张法线贴图
// 我们把B向量乘上手性系数翻转，得到相反的切线空间
vec3 B = cross(N, T);
mat3 TBN = mat3(T, B, N);
mat3 TBN_transpose = transpose(TBN);
```

但是需要注意的是：由于顶点平均化TBN向量时，实际上已经发生了数据信息的丢失。只要使用了平均化，我们不可能通过处理完全解决这个问题，只能尽可能的接近。

而且本质上，我们采用的方式是让T、B向量完全向N向量妥协，优先确保N向量的正确。

> 实际上，常见的3D模型会在平滑曲面等处才会使用类似的平滑化方法。而在模型的大幅转折处（比如立方体的棱角），会为每个三角形使用独立顶点来保证显示的正确性。

##### 非等比缩放

前面我们暂时忽略了非等比缩放，现在把它也加入讨论。

法向量N的处理我们很熟悉，将乘上模型矩阵改为乘上模型矩阵的逆转置矩阵。

对于T、B两个切向量，由于它们在平面上，我们依然让它们乘上模型矩阵。

此时会发现，虽然N依然垂直T、B平面，但T、B之间不再垂直。这是由于非等比缩放和旋转一起作用时，将平面拉伸导致的。

我们可以采用同样的格拉姆施密特正交化来进行处理。

此时我们得到的最终TBN矩阵中，N和T理论上并没有变化，而B则向T、N屈服。

这种处理也并不能完全解决非等比缩放的问题，在非等比缩放极其严重的情况下，由于B与实际上的贴图v轴差别太大，也会产生明显的瑕疵。

### 视差贴图

法线贴图实际上只能模拟一个表面上幅度不大的凹凸变化。也无法做到凹凸表面之间的遮挡关系。

我们可以这样理解，一个平面渲染在屏幕上时，法线贴图影响了每一个片段处的法向量朝向。让这个片段对应的平面部分发生了偏转。那么也意味着，法线贴图对平面凹凸的影响实际上只有一个片段大小的范围。无法产生更深的凹凸效果。

我们可以采用另一个技术：视差贴图（视差映射）。

视差贴图的原理很简单。我们在贴图里存储平面上的高度值，理论上，我们在计算光照时有足够的信息计算出每个片段的值。

![Diagram of how parallax mapping works in OpenGL](https://learnopengl.com/img/advanced-lighting/parallax_mapping_plane_height.png)

我们假定红线是我们希望物体的实际表面，而下面的是实际的渲染平面。

当我们渲染A处的片段时，我们实际上看到的应该是B点。我们希望能够在此时对其他贴图进行采样时，采用B点对应的平面上的纹理坐标，来模拟我们此时看到的是B点。

![Diagram of how parallax mapping works in OpenGL with vector scaled by fragment's height.](https://learnopengl.com/img/advanced-lighting/parallax_mapping_scaled_height.png)

我们可以采用一个近似的方法来求B点的近似。我们取A点的高度值H(A)，将视野方向的向量长度缩放为向量P。我们取P向量末尾点P对应的纹理坐标，把它作为B的近似。

实际上，这是一种相当粗略的做法，在高度变化剧烈的地方会产生较明显的瑕疵。但它具有一定的可行性。

#### 一些处理

如果我们在世界空间中进行这种处理，会遇到一些问题。主要出在如何获取P点对应的纹理坐标。我们可能会想到：已知世界空间中的当前平面，又知道以平面上点A为起点的向量P，我们可以将P投影到平面上，然后求重心坐标的方式得到P的纹理坐标。这种方法可行，但是并不简单。

我们同样可以使用切线空间来简化这一个过程，在切线空间中，P点坐标的xy就是对应纹理坐标的uv。

另外一点，我们不会在位移贴图（实际上是一个深度贴图）里存储其高度值，而是存储其深度值。在平坦表面上模拟深度比模拟高度更容易，会带来很多计算上的好处。

![Parallax mapping using a depth map instead of a heightmap](https://learnopengl.com/img/advanced-lighting/parallax_mapping_depth.png)

现在，我们不考虑一个平面向上凸起，而是平面凹陷进去。

此时依然有平面上的点A和应该看到的点B，我们可以使用类似的处理。取A的深度值H(A)然后反向缩放向量V得到P的纹理坐标，作为B的近似。

```c++
//vert和法线贴图相同


//frag
[...] 

uniform sampler2D normalMap;//法线贴图可以一起使用
uniform sampler2D depthMap;

uniform float height_scale;
// 这个变量被引入进来进行额外控制 
// 我们并没有确定H(A)值对应的向量实际长度（实际上应该与希望的凹陷深度有关，但是我们都缩放至[0,1]） 我们可以将1.0深度视为单位向量，此时dir向量单位化后乘上深度值正好为所需的P向量。但这样通常导致太过剧烈的视差效果。我们引入这个scale值来确保合理的显示效果。

vec2 ParallaxMapping(vec2 texCoords, vec3 viewDir);

void main()
{           
    // 切线空间下的视野方向向量
    vec3 viewDir   = normalize(fs_in.TangentViewPos - fs_in.TangentFragPos);
  
  	// 此处通过该片段的纹理坐标和视野方向向量计算 返回P的纹理坐标
    vec2 texCoords = ParallaxMapping(fs_in.TexCoords,  viewDir);

    
    vec3 diffuse = texture(diffuseMap, texCoords);
    vec3 normal  = texture(normalMap, texCoords);
    normal = normalize(normal * 2.0 - 1.0);
    [...]    
}

vec2 ParallaxMapping(vec2 texCoords, vec3 viewDir)
{ 
  	// 获取片段纹理的深度值
    float height =  texture(depthMap, texCoords).r;
  	
  	// 只需要计算p向量的xy轴即可
    vec2 p = viewDir.xy * (height * height_scale);
  	/*
  	如果按照先前推导其实应该这样实现：
  	vec3 p = viewDir * (height * height_scale);
  	return texCoords - p.xy;
    但我们并不需要真正缩放整个向量，只需要缩放其xy轴即可
  	*/
  	
  	// 这里计算的p朝向是朝向视野方向的 (-p)就是实际的偏移量
    return texCoords - p;    
}
```

实际上，我们计算p向量更经常使用另一种近似方法。

```c++
vec2 p = viewDir.xy / viewDir.z * (height * height_scale);
// 其实写为(viewDir.xy / viewDir.z * height) * height_scale更加合理
// 实际上，这里加上/viewDir.z后相当于利用相似三角形求解偏移量，而点P变成了深度为height的平面与向量p的交点。这种处理可以使得带有角度看向平面时的显示效果更佳。
// 这种使用/ viewDir.z的做法被称为Offset Limit
```

由于我们进行了纹理的偏移，在图形的边缘会出现超过纹理取值范围的纹理坐标。此时可能会因为纹理环绕方式而产生瑕疵。对于一个单平面来说，我们可以采用简单的处理方式：直接抛弃这些像素。

```c++
texCoords = ParallaxMapping(fs_in.TexCoords,  viewDir);
if(texCoords.x > 1.0 || texCoords.y > 1.0 || texCoords.x < 0.0 || texCoords.y < 0.0)
    discard;
```

对于一个闭合的三维模型，直接抛弃可能会产生问题。这时候我们可以采用设置纹理环绕方式为GL_REPEAT来解决。

#### 改进思路

我们当前的视差贴图在较大的角度，或者陡峭（纹理深度变化快）的地方会出现明显的瑕疵。

这是由于我们的使用的近似方式非常粗略。而在快速变化的地方，需要精细的计算才能产生正确的结果。

深度图本质上是一个离散的二元函数，它可以看作我们实际上希望的物体表面上的一个采样。理论上，我们可以求出函数的近似解，然后将视线向量作为直线求两者的交点。这样求出的交点则是我们认为最符合实际的交点。

为了性能考虑，我们不可能真的去求数学上的实际交点，但是它是我们接下来的各种近似方案的基本思路。

#### 陡峭视差映射

![img](https://learnopengl-cn.github.io/img/05/05/parallax_mapping_steep_parallax_mapping_diagram.png)

我们原先求视线V与物体表面的实际交点B的方法比较过于粗略，我们可以用这样一种数学上更加准确的求法：

我们取视线与表面交点为T0，记录其深度值。接着我们按一定步长增加深度，去求视线与对应深度层的交点。我们比较这些交点与T0的深度值。当我们找到第一个深度比T0浅的交点时（比如图上的T3），我们将T3的对应的纹理坐标作为该片段实际的纹理坐标。

对于第一个深度比T0浅的交点（此处为T3），物体的表面在其上方。对于一个连续函数，根据零点存在定理，我们知道真正的交点在T3之前，在T2之后。于是我们不再继续计算下一层，将T3的xy作为纹理坐标返回。

```c++
// 替换原本的视差函数
vec2 ParallaxMapping(vec2 texCoords, vec3 viewDir)
{ 
    const float numLayers = 10;// 我们采样的层数
    float layerDepth = 1.0 / numLayers;// 每一层的深度s

    float currentLayerDepth = 0.0;
  
    // 这是完整的P向量，height为1.0可以省去
    vec2 P = viewDir.xy * height_scale;
  	// 实际上依然推荐使用vec2 P = viewDir.xy / viewDir.z * height_scale;
  	// 这种情况下的P才是符合上述推导的，末尾为深度1.0平面交点的向量
    vec2 deltaTexCoords = P / numLayers;

		vec2  currentTexCoords     = texCoords;
		float currentDepthMapValue = texture(depthMap, currentTexCoords).r;

		while(currentLayerDepth < currentDepthMapValue)
		{
    		// 我们每一步为纹理坐标加上偏移量
    		currentTexCoords -= deltaTexCoords;
    		// 然后进行取值
    		currentDepthMapValue = texture(depthMap, currentTexCoords).r;  
    		// 然后步进
    		currentLayerDepth += layerDepth;  
		}

		return currentTexCoords;
}
```

不难想象出，这种算法的显示效果好坏与采样的层数直接相关。采样层数越多效果越好。但是采样层数同样会增加性能消耗，过高的层数会影响渲染速度。

我们会发现，当我们面对一个平面时，实际上不需要很大的样本数量就能实现不错的效果。因此我们可以使用一个自适应的样本数量来缓解这个问题。

```c++
const float minLayers = 8;
const float maxLayers = 32;
float numLayers = mix(maxLayers, minLayers, abs(dot(vec3(0.0, 0.0, 1.0), viewDir)));
```

#### 视差遮蔽映射

如果我们直接使用上述的方案去实现视差贴图，会发现一个非常明显的瑕疵：物体出现很明显的锯齿效果以及图层之间的断层。

这些瑕疵形成的原因是一样的：我们之前直接将T3(第一个较浅点)作为纹理点，而T3是按照深度步长取得的近似点。由于我们进行的是分层的采样，在T0点附近的其他片段计算时，他们得到的最终交点也在T3附近。而在距离一定距离后，会发生突变，之后一段范围的交点会在T4所在的层数。这造成了直接的分层现象。

解决方法也并不困难：我们之前发现了根据零点存在定理，交点应该在第一个较浅点之前（T3与T2之间）。

![img](https://learnopengl-cn.github.io/img/05/05/parallax_mapping_parallax_occlusion_mapping_diagram.png)

我们根据这两点的实际深度与采样深度进行线性插值，得到点Tp。此时的Tp依然为近似值，但是我们可以发现它比直接取T3要精确的多。

```c++
[...] // 计算出第一个较浅点

// 上一个步长的纹理坐标用于插值
vec2 prevTexCoords = currentTexCoords + deltaTexCoords;

// 计算两个深度差值
float afterDepth  = currentDepthMapValue - currentLayerDepth;
float beforeDepth = texture(depthMap, prevTexCoords).r - currentLayerDepth + layerDepth;

// 进行简单的线性插值即可
float weight = afterDepth / (afterDepth - beforeDepth);
vec2 finalTexCoords = prevTexCoords * weight + currentTexCoords * (1.0 - weight);

return finalTexCoords;
```

这种做法不仅能够更加精确的计算近似交点，而且由于插值的特性，采样层之间过渡会变得平滑。避免的分层的现象。

#### 浮雕视差映射

在进行陡峭视差映射的时候，我们可能会想到。这样一步步的步进求解，效率实际上并不高。如果我们采用二分法，第一次求0.5深度进行比较……，这样同样的采样次数我们可以得到更精确的解。

这看起来是可行的，但是存在一个致命的问题：它无法处理多个交点。

![img](https://pica.zhimg.com/v2-1b48f5b570444b077179e63dbed38f32_1440w.jpg)

当存在多个交点时，陡峭视差映射由于慢慢深入，找到的第一个点一定是第一个交点。而二分法无法确认找到的点是否为第一交点。

但这不意味这这种思路对我们视差贴图的实现没有帮助。

![img](https://pica.zhimg.com/v2-70e1c702f963518ab3fbcb74c6b70c60_1440w.jpg)

如图上所示，我们先进行陡峭视差映射的计算。在找到第一个较低浅点后，在两点之间（图中的2和3之间），进行二分法计算。找到更加接近于实际交点的那个点。

可以发现浮雕视差映射实际上也是陡峭视差映射的改进版本，是用于解决直接取第一个较浅点时的误差过大的问题。

与视差遮蔽映射不同的是，最后的精度完全取决于二分的迭代次数。迭代次数足够多的情况下，会比视差遮蔽更加准确。但由于遮蔽的算法采用了插值，只需要一步计算，计算效率会比浮雕更好。

#### 混合方案

实际上，我们可以发现两种对于陡峭视差映射的改良方法针对的部分并不相同。

POM假定最后两个采样点之间是近似直线的曲线，然后通过插值取得近似点。

Relief延续了陡峭视差映射取第一个较浅点的做法，但是通过二分法取得了更高的精度。

我们实际上可以混合两种方法，采用Relief最后取得的两个采样点进行POM的插值。由于两个采样点非常接近，此时的曲线实际上和两点间的线段非常接近。我们最后可以得到非常精确的交点。

这是一种追求极致显示效果的方案，需要进行大量的迭代和纹理采样。实际的实时渲染应用中很少会采用如此高性能消耗的做法来追求并不多的画面提升。

#### 自阴影

视差映射反应的是明显的深度关系，这种明显的深度关系中通常包含着大量的遮挡关系。阴影对于遮挡关系的表现有着非常好的效果。能够极大的加强我们视差贴图的显示效果。

在先前的视差映射算法中，我们通过从摄像机发射视线向量 $V$ 来寻找其与物体表面的交点。

计算自阴影的思路与寻找交点类似，它被称为**反向光线步进（Reverse Raymarching）**：我们不再从摄像机出发，而是**以刚刚算出的交点为起点，沿着光线的反方向（即指向光源的向量 $L$）发射一条射线**。

如果这条射线在飞出深度为 0.0 的表面之前，撞到了任何比它“高”的表面（即深度图采样值小于光线的当前深度），就说明光线被地形挡住了，这个点就处于阴影之中。

我们先来实现一个硬阴影的方案，只考虑当前的片段是否被遮住了：

```c++
// frag
uniform float height_scale;
uniform sampler2D depthMap;

// lightDir 是切线空间下，从片段指向光源的向量 (已标准化)
// initialTexCoords 是 POM 算出的最终纹理坐标
// initialDepth 是 POM 算出交点时的深度值
float ParallaxShadow(vec3 lightDir, vec2 initialTexCoords, float initialDepth)
{
    // 如果光源在平面的下方，直接认为全黑
    if(lightDir.z < 0.0)
        return 0.0;

    // 自适应步数：光线越平缓（掠射角），需要的层数越多，因为更容易发生遮挡
    const float minLayers = 15.0;
    const float maxLayers = 30.0;
    float numLayers = mix(maxLayers, minLayers, abs(lightDir.z));

    // 计算每一步的深度步长
    // 因为我们要从 initialDepth 走到 0.0，所以总路程是 initialDepth
    float layerDepth = initialDepth / numLayers;

    // 计算每一步的纹理坐标偏移量
    // 与求交点类似，利用相似三角形，沿着光源方向偏移
    vec2 deltaTexCoords = (lightDir.xy / lightDir.z) * height_scale * layerDepth;

    // 初始化起点
    float currentLayerDepth = initialDepth - layerDepth;
    vec2 currentTexCoords = initialTexCoords + deltaTexCoords;

    // 向上步进，直到离开表面 (深度 <= 0.0)
    while(currentLayerDepth > 0.0)
    {
        // 获取当前纹理坐标处的地势深度
        float currentDepthMapValue = texture(depthMap, currentTexCoords).r;

        // 如果地势比射线当前的高度更高（深度值更小），发生遮挡
        if(currentDepthMapValue < currentLayerDepth)
        {
            return 0.0; // 直接返回 0.0 产生硬阴影
        }

        // 继续向着光源步进
        currentLayerDepth -= layerDepth;
        currentTexCoords += deltaTexCoords;
    }

    return 1.0; // 成功逃逸，没有被遮挡
}
```

为了实现软阴影，我们需要考虑其他因素。真实的软阴影是由于光源具有体积，在完全不被照射和完全被照射到的区域之间有一块不完全照射的半影区。在处理软阴影时，我们通常不会直接去计算半影区本身。而是通过平滑化光到影的过渡区域实现的。

我们可以这样考虑：一个片段在阴影中的程度，和它被遮挡住的程度相关。也就是说，一个片段与光源之间如果间隔的物体很矮，那么它的阴影就会更浅。如果间隔的物体很高，阴影就会更深。

如何判断与光源间阻隔物体的高度呢？我们可以进行一个近似。我们不再一遇到深度更浅的点就直接返回，而是计算当前射线步进到到深度和实际深度的差值。我们取所有差值中最大的作为该片段的遮挡程度计算。

```c++
float ParallaxSoftShadow(vec3 lightDir, vec2 initialTexCoords, float initialDepth)
{
    if(lightDir.z < 0.0) return 0.0;

    const float minLayers = 15.0;
    const float maxLayers = 30.0;
    float numLayers = mix(maxLayers, minLayers, abs(lightDir.z));

    float layerDepth = initialDepth / numLayers;
    vec2 deltaTexCoords = (lightDir.xy / lightDir.z) * height_scale / numLayers;

    float currentLayerDepth = initialDepth - layerDepth;
    vec2 currentTexCoords = initialTexCoords + deltaTexCoords;

    // 记录最大的遮挡值
    float shadowFactor = 0.0; 

    // 控制阴影的柔和程度，值越大阴影越硬
  	// 该算法下，所有片段的遮挡程度实际上只能取到<1.0的值
  	// 我们通过这个值将其放缩，使得遮挡程度大于某个值时，就视为完全处于阴影（返回值为0.0）
    const float shadowHardness = 4.0; 
  
  	// 距离起点的累计深度（用于计算距离）
		// float distanceTraveled = 0.0;

    while(currentLayerDepth > 0.0)
    {
        float currentDepthMapValue = texture(depthMap, currentTexCoords).r;

        if(currentDepthMapValue < currentLayerDepth)
        {
            // 计算当前步的遮挡程度：差值越大，地形比光线高的越多
            float currentOcclusion = (currentLayerDepth - currentDepthMapValue);
          	
          	// 实际上更加符合物理的情况应该是在加入被遮挡的距离
          	// 如果很近处就被遮挡，那么阴影也会加深
        		// float softFactor = currentOcclusion / distanceTraveled;
          	// shadowFactor = max(shadowFactor, softFactor);
          	// 这种情况下shadowHardness也需要更改为合适的值
          
            // 保留最大的遮挡程度
            shadowFactor = max(shadowFactor, currentOcclusion);
        }

        currentLayerDepth -= layerDepth;
        currentTexCoords += deltaTexCoords;
      	// distanceTraveled += layerDepth;  累加走过的距离
    }

    // 将遮挡因子转换为最终的光照倍率
    // shadowFactor 越大，结果越接近 0.0（纯黑）
    return clamp(1.0 - shadowFactor * shadowHardness, 0.0, 1.0);
}
```

我们最后只需要将阴影直接加入最后的光线计算即可：

```c++
// 获取 POM 算出的最终纹理坐标和深度
// 采样基础纹理和法线

// 计算自阴影乘数
float shadowMult = ParallaxSoftShadow(lightDir, finalTexCoords, outDepth);

// 常规 Blinn-Phong 计算

// 只有漫反射和镜面反射部分乘以阴影乘数
vec3 finalColor = ambient + (diffuseLight + specularLight) * shadowMult;
```

#### 与阴影映射一起使用

实际上，自阴影处理的阴影与阴影映射是完全不同维度的问题。

自阴影的遮挡完全来自其自身，而阴影映射的遮挡来自于其他物体。所以我们可以同时使用两种阴影，只需要在最后同时应用两个阴影计算得出的系数即可。

#### 实际应用的问题

我们已经实现了效果不错的视差映射效果。但是在实际工程应用中，存在一些与我们理论无关的问题会影响到显示效果。其中最主要的问题是**Mipmap 瑕疵**。

在我们采样的时候，直接使用texture函数对纹理进行了采样，无论我们使用的是不是mipmap贴图。那么计算机是如何确认该使用什么层级的mipmap的呢？

答案在于屏幕空间偏导数（Screen-space Derivatives）和 **$2 \times 2$ 像素块（Fragment Quad）**。

在光栅化阶段，GPU 执行片元着色器时，**并不是**一个像素一个像素孤立执行的，而是永远以 $2 \times 2$ 的像素块（Quad）为最小调度单位进行渲染。即使三角形只盖住了其中一个像素，另外三个像素也会在后台被计算（只是最终会被丢弃），这是为了计算偏导数。

在计算当前像素的纹理采样时，GPU 会获取它旁边像素的 UV 坐标并计算出差值：dFdx(uv)和dFdy(uv)。

我们的采样函数认为：当uv的变化很小时，说明我们正在近距离观看一个平面，它会选择最低层级的mipmap来进行采样(Mipmap Level 0)。而当uv变化很大时，则说明平面离我们很远，则采样更模糊层级更高的mipmap。对于一般的贴图来说，这种策略是正确的。

但是这种策略在视差贴图上会产生很大的问题。原因在于：视差贴图是用来表示物体表面的凹凸性的，在深度快速变化处，我们计算出的uv变化也很大。直接使用texture函数采样会导致gpu认为这些地方应该选择高层级的mipmap，从而导致采样出的结果失真。

为了修正这个问题，我们需要手动处理mipmap的层级。

```c++
// 在进行POM之类的计算之前 我们可以手动计算dx和dy
// 此时的 TexCoords 是平滑的线性插值，算出的 dx/dy 完美代表了模型的真实远近和角度
vec2 dx = dFdx(fs_in.TexCoords);
vec2 dy = dFdy(fs_in.TexCoords);

// POM计算 得到最后的finalTexCoords
// 为了保证计算的精确性，我们需要强制让所有计算都采样最低的mipmap层级
// 可以使用textureLod实现
// float currentDepthMapValue = textureLod(depthMap, currentTexCoords, 0.0).r;

// textureGrad可以让我们使用自己的dx，dy来指定mipmap层级
vec3 diffuse = textureGrad(diffuseMap, finalTexCoords, dx, dy).rgb;
```
