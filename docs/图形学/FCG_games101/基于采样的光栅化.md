光栅化的过程，可以理解为将顶点着色器传递下来的图元（通常为三角形）绘制在屏幕上的过程，每个被绘制的片段（像素）会被执行片段着色器的阶段。

在早期（比如上世纪 80-90 年代，早期软件渲染和早期的 3D 游戏时代），计算资源极其宝贵。扫描线算法是当时的绝对主流的光栅化算法。

### 扫描线算法

**1. 它是如何工作的？** 算法顾名思义，是逐行扫描屏幕的。它用一条水平线（扫描线）从屏幕上方往下扫，计算这条线与三角形边缘的交点。在两个交点之间的连续区域，就被认为是三角形的内部，然后将其填充为相应的颜色。

**2. 为什么过去要用它？**

- **计算极其节省（利用空间连贯性）：** 扫描线算法极大地利用了“连贯性（Coherence）”。如果当前像素在三角形内，它旁边的像素大概率也在。通过简单的增量加法（Bresenham 画线算法的变种），它能用极少的计算量推导出下一个像素的状态，避免了复杂的乘法计算。
- **内存占用小：** 在没有深度缓冲（Z-Buffer）或者 Z-Buffer 极其昂贵的年代，扫描线算法可以结合多边形排序（画家算法），做到只渲染可见部分，节省内存读取。

**3. 致命弱点（为什么被淘汰）：**

- **串行依赖极强：** 它必须一行一行算，每一行的计算依赖上一行的状态（比如边缘数据结构 AET：Active Edge Table）。**这种串行逻辑几乎无法被现代 GPU 的并行架构所利用。**

### 基于采样的光栅化

现代图形学换了一个思路：不再试图“聪明地”寻找边界并填充，而是采用一种看似“笨重”但在数学上极其优雅的方法——**采样**。

**1. 它是如何工作的？**

对于屏幕上的每一个像素，我们在其中心点（或多个子采样点）进行一次测试：“**这个点是否在三角形内部？**”

通常使用边缘方程（Edge Equations）或叉乘来实现：对于三角形的三条边，判断该像素点是否同时在三条边的“内侧”。如果是，这个像素就被照亮。

通常使用边缘方程（Edge Equations）或叉乘来实现：对于三角形的三条边，判断该像素点是否同时在三条边的“内侧”。如果是，这个像素就被照亮。

**2. 为什么现代 GPU 全面拥抱采样算法？**

这是现代图形硬件设计的核心所在：

- **极致的并行化（最重要的原因）：** 采样算法中，像素 A 是否在三角形内，与像素 B 是否在三角形内**毫无关系**。这意味着 GPU 可以派出成千上万个计算核心，同时对成百上千个像素进行独立的“点在多边形内”测试。这种“暴力而独立”的特性，完美契合了 GPU 的 SIMD（单指令多数据流）架构。
- **硬件实现的简单性：** 边缘方程本质上只需要做基础的乘法和加法（$Ax + By + C > 0$）。硬件设计者非常喜欢这种不需要维护复杂状态机、不需要查表（无需 AET 表）的纯数学计算单元。
- **优雅的属性插值（重心坐标）：** 在判断一个点是否在三角形内部的同时，采样算法的副产物直接就是**重心坐标（Barycentric Coordinates）**。现代管线（Shader）极度依赖重心坐标来对顶点属性（颜色、法线、UV 贴图坐标）进行平滑插值，这在采样算法中是顺理成章的。
- **抗锯齿（MSAA）的完美契合：** 现代游戏需要多重采样抗锯齿（Multi-Sample Anti-Aliasing）。如果是基于采样的光栅化，只需在一个像素内设置 4 个或 8 个采样点，独立进行测试即可，逻辑异常简单。而传统的扫描线算法处理亚像素级别的抗锯齿会变得极其复杂和低效。
