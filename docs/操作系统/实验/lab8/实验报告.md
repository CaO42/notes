## Lab8 从内核态到用户态

### 实验环境

- 操作系统版本：虚拟机ubuntu arm64 22.04LTS
- CPU 架构：ARM64
- 主要工具：QEMU、QEMU Monitor、gdb/gdb-multiarch、gcc/g++ 或 i686-elf-gcc/g++、nasm、ld/i686-elf-ld

### 实验概述

本次实验围绕保护模式下的特权级转换、系统调用、用户进程创建、`fork`、`wait` 和 `exit` 等进程生命周期机制展开。

在前面的实验中，内核线程都运行在 0 特权级，线程之间共享同一套内核地址空间。本实验进一步引入 3 特权级用户态，让普通程序运行在受限环境中。用户程序不能直接访问内核数据段、内核栈和特权指令，当它需要使用内核服务时，需要通过系统调用进入内核态。

系统调用本质上是一种受控的特权级转换。本实验使用 `int 0x80` 软中断作为系统调用入口：用户态代码先把系统调用号和参数放入寄存器，然后触发中断；CPU 根据 IDT 中的中断描述符进入内核代码段，并在特权级提升时从 TSS 中取得 0 特权级内核栈地址；内核处理完成后通过 `iret` 恢复用户态现场。

在此基础上，实验继续实现用户进程。进程相比内核线程多出了独立的用户虚拟地址空间、页目录表、用户栈和父子关系等信息。调度器在切换进程时不仅要切换线程栈，还要切换页目录表寄存器 `CR3`，从而实现进程间地址空间隔离。

最后，实验实现了 `fork`、`exit` 和 `wait`。`fork` 用于复制当前进程并创建子进程，父进程和子进程从同一个系统调用返回点继续执行，但返回值不同；`exit` 负责保存进程退出状态、释放用户空间资源并让出 CPU；`wait` 则用于父进程回收已经退出的子进程，避免 PCB 长期处于无法释放的僵尸状态。

本次实验主要任务如下：

- 实验任务1：复现系统调用机制，添加自定义系统调用，分析 `asm_system_call`、`int 0x80`、TSS 栈切换、`asm_system_call_handler` 和 `iret` 的完整执行流程，并通过 GDB 或 QEMU Monitor 观察特权级切换前后的寄存器变化。
- 实验任务2：复现用户进程创建机制，分析 PCB 扩展、用户页目录表、`ProcessStartStack`、`asm_start_process` 和调度时 `CR3` 切换；进一步复现 `fork`，分析子进程地址空间复制、用户空间复制、内核栈构造以及父子进程返回值差异。
- 实验任务3：复现 `exit` 和 `wait`，分析进程资源释放、退出状态保存、父进程回收子进程 PCB 的过程，并对僵尸进程和孤儿进程问题给出处理方案。

### A1 系统调用与特权级转换

#### A1.1 自定义系统调用复现

系统调用是用户程序访问内核服务的入口。用户程序运行在 3 特权级，不能直接调用内核中的特权代码，因此需要通过一个受控入口进入 0 特权级。

本实验在原有 `syscall_0` 的基础上，计划新增两个自定义系统调用：

- `syscall_max`：接收两个整数，返回二者中的较大值。
- `syscall_factorial`：接收一个非负整数，使用迭代方式返回其阶乘值。

系统调用的注册过程仍然通过 `SystemService::setSystemCall` 完成。每个系统调用号对应系统调用表 `system_call_table` 中的一个函数地址。用户态封装函数不直接调用内核函数，而是调用 `asm_system_call`，由汇编代码把系统调用号放入 `eax`，把参数依次放入 `ebx`、`ecx`、`edx`、`esi`、`edi`，然后执行 `int 0x80`。

关键代码可以分为“系统调用声明”“用户态包装函数”“内核态处理函数”“系统调用表注册”和“测试代码”五部分。

首先在 `include/syscall.h` 中加入新增系统调用的声明。这里将系统调用号 5 用作求最大值，系统调用号 6 用作阶乘。

```c++
// 第5个系统调用，返回两个整数的最大值
int get_max(int first, int second);
int syscall_get_max(int first, int second);

// 第6个系统调用，计算阶乘
int factorial(int n);
int syscall_factorial(int n);
```

然后在 `src/kernel/syscall.cpp` 中加入用户态包装函数。用户程序不直接调用 `syscall_get_max` 和 `syscall_factorial`，而是通过 `asm_system_call` 触发 `int 0x80`。

```c++
int get_max(int first, int second)
{
    return asm_system_call(5, first, second);
}

int factorial(int n)
{
    return asm_system_call(6, n);
}
```

系统调用对应的内核处理函数可以放在 `src/kernel/setup.cpp` 中，和已有的 `syscall_0` 放在一起。

```c++
int syscall_get_max(int first, int second)
{
    return first > second ? first : second;
}

int syscall_factorial(int n)
{
    if (n < 0)
    {
        return -1;
    }

    int result = 1;
    for (int i = 2; i <= n; ++i)
    {
        result *= i;
    }

    return result;
}
```

在 `setup_kernel` 初始化系统调用表时注册这两个系统调用。

```c++
systemService.initialize();
systemService.setSystemCall(0, (int)syscall_0);
systemService.setSystemCall(5, (int)syscall_get_max);
systemService.setSystemCall(6, (int)syscall_factorial);
```

最后，在用户进程或 `first_thread` 中调用新增系统调用并打印结果。若使用 `src/3` 及之后的用户进程框架，可以放在 `first_process` 中测试。

```c++
void first_process()
{
    int first = 17;
    int second = 42;
    int n = 5;

    int maxValue = get_max(first, second);
    int factorialValue = factorial(n);

    printf("syscall get_max(%d, %d) = %d\n", first, second, maxValue);
    printf("syscall factorial(%d) = %d\n", n, factorialValue);

    asm_halt();
}
```

如果在 `src/1` 或 `src/2` 中直接基于 `first_thread` 测试，也可以把上面的调用放入 `first_thread`。核心不变，区别只是调用位置不同。

#### 系统调用执行流程

系统调用的完整过程可以分为用户态准备、特权级转移、内核态分发和中断返回四个阶段。

第一阶段是用户态准备。用户程序调用类似 `asm_system_call(index, first, second, third, forth, fifth)` 的包装函数后，汇编代码先保存必要的寄存器，再把系统调用号放入 `eax`，把最多五个参数分别放入 `ebx`、`ecx`、`edx`、`esi`、`edi`。之所以使用寄存器传参，是因为系统调用会从用户栈切换到内核栈。如果参数留在用户栈中，进入内核态后 C/C++ 函数按照当前 `esp`、`ebp` 访问的已经是内核栈，无法直接按普通调用约定取得原来的参数。

第二阶段是 `int 0x80` 触发特权级转移。`SystemService::initialize` 中将 0x80 号中断描述符的 DPL 设置为 3，使得 CPL=3 的用户态程序允许调用该中断。中断门中保存的代码段选择子指向 0 特权级代码段，因此 CPU 在执行 `int 0x80` 后会从用户态切换到内核态。

当中断造成特权级从 3 提升到 0 时，CPU 会从当前 TSS 中取出 `ss0` 和 `esp0`，将它们加载到 `SS` 和 `ESP`，从而切换到当前进程对应的内核栈。随后 CPU 会把中断发生前的用户态 `SS`、`ESP`、`EFLAGS`、`CS`、`EIP` 依次压入新的内核栈。这些内容就是之后 `iret` 返回用户态所需的现场。

第三阶段是内核态系统调用分发。`asm_system_call_handler` 进入后会保存段寄存器和通用寄存器，重新设置 `ds`、`es`、`gs` 等段寄存器，使内核代码可以访问内核数据段和显存段。然后它把寄存器中的系统调用参数压入内核栈，根据 `eax` 中的系统调用号索引 `system_call_table`，调用对应的内核服务函数。

系统调用处理函数返回后，返回值保存在 `eax` 中。由于后续需要恢复通用寄存器，处理程序会先把返回值临时保存到 `ASM_TEMP`，待 `popad` 和段寄存器恢复完成后，再把返回值放回 `eax`。

第四阶段是 `iret` 返回用户态。`iret` 会从当前内核栈中弹出先前保存的 `EIP`、`CS`、`EFLAGS`。如果返回时发生从 0 特权级到 3 特权级的转换，CPU 还会继续弹出 `ESP` 和 `SS`，恢复用户态栈。最终 CPU 回到用户态 `int 0x80` 后的下一条指令继续执行，此时系统调用返回值已经位于 `eax`。

#### TSS 在系统调用中的作用

TSS 的核心作用并不是保存本实验中每个进程的完整上下文，而是为 CPU 提供特权级转换时要使用的高特权级栈。

本实验中真正用到的是 TSS 的两个字段：

- `ss0`：0 特权级内核栈段选择子。
- `esp0`：当前进程进入内核态时应使用的内核栈栈顶。

`initializeTSS` 会初始化 TSS 描述符，并通过 `ltr` 指令把 TSS 选择子加载到任务寄存器 `TR` 中。之后每当用户态通过中断进入内核态时，CPU 就能够自动从 TSS 中找到内核栈。

调度器切换到用户进程时，`activateProgramPage` 会把 `tss.esp0` 设置为当前进程 PCB 所在页的顶部。这样每个用户进程都有自己的内核栈。用户进程执行系统调用或中断进入内核时，会使用自己 PCB 页中的内核栈保存现场，避免不同进程的内核态执行现场互相覆盖。

#### A1.2 GDB 或 QEMU Monitor 分析

本部分需要在 `src/3` 或对应进程示例中，在 `int 0x80` 前、进入 `asm_system_call_handler` 后、执行 `iret` 后观察寄存器变化。

> **截图位置 A1-2：`int 0x80` 前后的寄存器变化**  
> 建议文件：`./gallery/a1_registers_before_after_syscall.png`

| 观察位置 | CS | SS | ESP | EIP | CPL | 说明 |
| --- | --- | --- | --- | --- | --- | --- |
| 执行 `int 0x80` 前 | 待补充 | 待补充 | 待补充 | 待补充 | 3 | 此时仍在用户态，`CS` 的低两位为 3，`SS` 指向用户栈段。 |
| 进入 `asm_system_call_handler` 后 | 待补充 | 待补充 | 待补充 | 待补充 | 0 | CPU 已经根据 TSS 中的 `ss0`、`esp0` 切换到内核栈，`CS` 指向 0 特权级代码段。 |
| 执行 `iret` 后 | 待补充 | 待补充 | 待补充 | 待补充 | 3 | `iret` 从内核栈恢复用户态 `CS/EIP/SS/ESP/EFLAGS`，程序回到系统调用之后继续执行。 |

从理论上看，在 `int 0x80` 之前，当前 CPL 由 `CS` 选择子的低两位决定，应为 3；进入 `asm_system_call_handler` 后，CPU 使用中断描述符中的 0 特权级代码段选择子，因此 CPL 变为 0。新的 `SS` 和 `ESP` 并不是从用户程序中取得，而是来自当前 TSS 的 `ss0` 和 `esp0`。

在内核栈上，CPU 自动保存了返回用户态所需的信息，包括用户态 `SS`、`ESP`、`EFLAGS`、`CS` 和 `EIP`。因此 `iret` 可以在系统调用处理结束后恢复这些值，使进程回到原来的用户态执行位置。

### A2 进程创建与 Fork

#### A2.1 进程创建过程分析

用户进程可以看作在线程机制基础上增加了独立用户地址空间和用户态启动现场的执行流。内核线程只需要 PCB 和内核栈，而用户进程还需要页目录表、用户虚拟地址池、用户代码段/数据段/栈段选择子以及能够通过 `iret` 进入用户态的中断栈结构。

> **截图位置 A2-1：多个用户进程通过系统调用输出信息**  
> 建议文件：`./gallery/a2_multiple_processes.png`

本实验中的进程创建可以概括为三步。

第一步是在内核线程 PCB 的基础上扩展进程 PCB。`PCB` 中原有的 `stack`、`status`、`pid`、`ticks` 等字段仍然用于调度；为了支持进程，还新增了 `pageDirectoryAddress`、`userVirtual`、`parentPid` 和 `retValue` 等字段。

`executeProcess` 首先复用 `executeThread` 创建一个执行 `load_process` 的内核调度实体。随后它为该 PCB 创建独立页目录表，并创建用户虚拟地址池。线程的 `pageDirectoryAddress` 为 0，表示它运行在内核共享地址空间；进程的 `pageDirectoryAddress` 非 0，表示它拥有自己的用户页目录表。

关键代码如下。它先创建一个特殊线程，这个线程第一次运行时会执行 `load_process`，再补充分配进程专有的页目录表和用户虚拟地址池。

```c++
int ProgramManager::executeProcess(const char *filename, int priority)
{
    bool status = interruptManager.getInterruptStatus();
    interruptManager.disableInterrupt();

    int pid = executeThread((ThreadFunction)load_process,
                            (void *)filename, filename, priority);
    if (pid == -1)
    {
        interruptManager.setInterruptStatus(status);
        return -1;
    }

    PCB *process = ListItem2PCB(allPrograms.back(), tagInAllList);

    process->pageDirectoryAddress = createProcessPageDirectory();
    if (!process->pageDirectoryAddress)
    {
        process->status = ProgramStatus::DEAD;
        interruptManager.setInterruptStatus(status);
        return -1;
    }

    if (!createUserVirtualPool(process))
    {
        process->status = ProgramStatus::DEAD;
        interruptManager.setInterruptStatus(status);
        return -1;
    }

    interruptManager.setInterruptStatus(status);
    return pid;
}
```

创建页目录表时，内核会从内核地址池中分配一页作为页目录表，并把高 1GB 的内核空间页目录项复制到新页目录表中。这样所有进程虽然拥有独立的低 3GB 用户空间，但仍然共享高地址处的内核映射。页目录表最后一项指向页目录表自身，便于通过固定虚拟地址访问当前页表和页目录。

第二步是设置 `ProcessStartStack`，使进程能够通过 `asm_start_process` 和 `iret` 从内核态进入用户态。

`ProcessStartStack` 的布局模拟了一次中断返回所需要的栈帧。内核在该结构中填入用户态寄存器初值，包括：

- `eip`：用户进程入口函数地址。
- `cs`：3 特权级用户代码段选择子。
- `ds/es/fs/ss`：3 特权级用户数据段或用户栈段选择子。
- `eflags`：设置必要保留位，并打开中断。
- `esp`：新分配的用户栈栈顶。

`asm_start_process` 会把 `esp` 切换到这个人工构造的 `ProcessStartStack`，然后依次恢复通用寄存器、段寄存器，最后执行 `iret`。由于 `ProcessStartStack` 中的 `CS` 和 `SS` 都是 3 特权级选择子，所以 `iret` 后 CPU 进入用户态执行。

`load_process` 的关键在于人工构造这份中断返回栈。用户进程入口地址放入 `eip`，用户代码段选择子放入 `cs`，用户栈放入 `esp/ss`，最后调用 `asm_start_process`。

```c++
void load_process(const char *filename)
{
    interruptManager.disableInterrupt();

    PCB *process = programManager.running;
    ProcessStartStack *interruptStack =
        (ProcessStartStack *)((int)process + PAGE_SIZE - sizeof(ProcessStartStack));

    interruptStack->edi = 0;
    interruptStack->esi = 0;
    interruptStack->ebp = 0;
    interruptStack->esp_dummy = 0;
    interruptStack->ebx = 0;
    interruptStack->edx = 0;
    interruptStack->ecx = 0;
    interruptStack->eax = 0;
    interruptStack->gs = 0;

    interruptStack->fs = programManager.USER_DATA_SELECTOR;
    interruptStack->es = programManager.USER_DATA_SELECTOR;
    interruptStack->ds = programManager.USER_DATA_SELECTOR;

    interruptStack->eip = (int)filename;
    interruptStack->cs = programManager.USER_CODE_SELECTOR;
    interruptStack->eflags = (0 << 12) | (1 << 9) | (1 << 1);

    interruptStack->esp = memoryManager.allocatePages(AddressPoolType::USER, 1);
    if (interruptStack->esp == 0)
    {
        printf("can not build process!\n");
        process->status = ProgramStatus::DEAD;
        asm_halt();
    }

    interruptStack->esp += PAGE_SIZE;

    int *userStack = (int *)interruptStack->esp;
    userStack -= 3;
    userStack[0] = (int)exit;
    userStack[1] = 0;
    userStack[2] = 0;
    interruptStack->esp = (int)userStack;

    interruptStack->ss = programManager.USER_STACK_SELECTOR;

    asm_start_process((int)interruptStack);
}
```

第三步是在调度中切换页目录表。`ProgramManager::schedule` 选出下一个就绪程序后，会先调用 `activateProgramPage(next)`。如果 `next->pageDirectoryAddress` 非 0，说明下一个调度对象是用户进程，内核需要把该页目录表的物理地址写入 `CR3`。写入 `CR3` 后，CPU 使用新的页目录表完成地址翻译，用户进程之间的低地址空间因此相互隔离。

如果调度对象是内核线程，其 `pageDirectoryAddress` 为 0，调度器会把 `CR3` 恢复为内核页目录表地址。这样内核线程继续运行在内核共享地址空间中。

`activateProgramPage` 还会在切换到用户进程时更新 `tss.esp0`。这一步保证进程下一次从用户态进入内核态时，CPU 会使用当前进程自己的内核栈。

```c++
void ProgramManager::activateProgramPage(PCB *program)
{
    int paddr = PAGE_DIRECTORY;

    if (program->pageDirectoryAddress)
    {
        tss.esp0 = (int)program + PAGE_SIZE;
        paddr = memoryManager.vaddr2paddr(program->pageDirectoryAddress);
    }

    asm_update_cr3(paddr);
}
```

#### 为什么线程的 `pageDirectoryAddress` 为 0

内核线程只在 0 特权级运行，直接使用内核页目录表和内核地址空间，不需要独立的用户虚拟地址空间。因此它的 `pageDirectoryAddress` 被设置为 0，调度器可以据此判断该调度对象是否需要切换到用户页目录表。

用户进程则需要隔离自己的用户空间。每个进程的低 3GB 地址空间可以映射到不同的物理页，同一个虚拟地址在不同进程中可以表示不同数据。因此用户进程必须拥有非 0 的 `pageDirectoryAddress`。

#### A2.2 Fork 的实现与分析

`fork` 的目标是创建一个几乎和父进程完全相同的子进程。父子进程拥有相同的用户代码和数据内容，从同一个系统调用返回点继续执行，但它们处在不同的地址空间中，并且 `fork` 返回值不同：父进程返回子进程 pid，子进程返回 0。

父子进程测试代码如下。父进程中 `fork` 返回子进程 pid，子进程中 `fork` 返回 0，因此可以根据返回值区分两条执行路径。

```c++
void first_process()
{
    int ret = fork();

    if (ret < 0)
    {
        printf("fork failed\n");
        asm_halt();
    }

    if (ret == 0)
    {
        printf("child: pid=%d, fork return=%d\n",
               programManager.running->pid, ret);
    }
    else
    {
        printf("father: pid=%d, fork return child pid=%d\n",
               programManager.running->pid, ret);
    }

    asm_halt();
}
```

`fork` 的实现需要解决四个关键问题。

第一，如何为子进程创建独立虚拟地址空间。`fork` 首先调用进程创建逻辑，为子进程分配新的 PCB、页目录表和用户虚拟地址池。随后复制父进程用户空间部分的页目录项和页表项。内核高 1GB 空间仍然共享，因为这部分映射在所有进程中一致；用户低 3GB 空间则需要为子进程单独建立映射。

第二，如何复制父进程用户空间数据。实现时可以遍历父进程用户空间页目录项和页表项，找到已经存在的用户页。对每个用户页，从用户物理地址池中为子进程分配新的物理页，并把父进程该虚拟页中的内容复制到子进程对应物理页中。复制过程中需要注意当前 `CR3` 决定了虚拟地址解释方式，因此通常会借助一页内核缓冲区，在父子页目录之间切换并完成数据搬运。

第三，如何设置子进程内核栈，使它第一次被调度时能正确运行。子进程不是从 `ProgramManager::fork` 的函数开头执行，而是应该像父进程一样从系统调用返回路径回到用户态。因此需要把父进程当前的 `ProcessStartStack` 复制到子进程 PCB 页顶部，并把子进程 PCB 中的 `stack` 设置成一次线程切换后能够返回到 `asm_start_process` 的形态。

调度器第一次切换到子进程时，`asm_switch_thread` 会把 `esp` 切换到子进程的内核栈，随后通过 `ret` 跳转到 `asm_start_process`。`asm_start_process` 再根据子进程的 `ProcessStartStack` 恢复寄存器并执行 `iret`，最终让子进程回到用户态的 `fork` 返回点。

第四，如何保证父子进程 `fork` 返回值不同。父进程是正常执行 `ProgramManager::fork` 后返回，因此它的返回值可以直接设为新创建子进程的 pid。子进程并没有真正执行完这次 C++ 函数调用，而是使用复制来的中断返回栈返回用户态。因此只需要把子进程 `ProcessStartStack` 中保存的 `eax` 改为 0，`iret` 回到用户态后，子进程看到的系统调用返回值就是 0。

`fork` 的入口函数可以写成如下形式。内核线程没有用户页目录表，因此不允许调用 `fork`。

```c++
int ProgramManager::fork()
{
    bool status = interruptManager.getInterruptStatus();
    interruptManager.disableInterrupt();

    PCB *parent = this->running;
    if (!parent->pageDirectoryAddress)
    {
        interruptManager.setInterruptStatus(status);
        return -1;
    }

    int pid = executeProcess("", 0);
    if (pid == -1)
    {
        interruptManager.setInterruptStatus(status);
        return -1;
    }

    PCB *child = ListItem2PCB(this->allPrograms.back(), tagInAllList);
    if (!copyProcess(parent, child))
    {
        child->status = ProgramStatus::DEAD;
        interruptManager.setInterruptStatus(status);
        return -1;
    }

    interruptManager.setInterruptStatus(status);
    return pid;
}
```

`copyProcess` 是 `fork` 中最关键的部分。下面只列出核心片段，重点是复制方向：父进程的中断栈复制到子进程，父进程的用户虚拟地址池位图复制到子进程，父进程用户页内容先复制到内核缓冲区，再切换到子进程地址空间复制进去。

```c++
bool ProgramManager::copyProcess(PCB *parent, PCB *child)
{
    ProcessStartStack *childpss =
        (ProcessStartStack *)((int)child + PAGE_SIZE - sizeof(ProcessStartStack));
    ProcessStartStack *parentpss =
        (ProcessStartStack *)((int)parent + PAGE_SIZE - sizeof(ProcessStartStack));

    // 注意复制方向：从父进程复制到子进程
    memcpy(childpss, parentpss, sizeof(ProcessStartStack));
    childpss->eax = 0;

    child->stack = (int *)childpss - 7;
    child->stack[0] = 0;
    child->stack[1] = 0;
    child->stack[2] = 0;
    child->stack[3] = 0;
    child->stack[4] = (int)asm_start_process;
    child->stack[5] = 0;
    child->stack[6] = (int)childpss;

    child->status = ProgramStatus::READY;
    child->parentPid = parent->pid;
    child->priority = parent->priority;
    child->ticks = parent->ticks;
    child->ticksPassedBy = parent->ticksPassedBy;
    strcpy(child->name, parent->name);

    int bitmapLength = parent->userVirtual.resources.length;
    int bitmapBytes = ceil(bitmapLength, 8);
    memcpy(child->userVirtual.resources.bitmap,
           parent->userVirtual.resources.bitmap,
           bitmapBytes);

    char *buffer = (char *)memoryManager.allocatePages(AddressPoolType::KERNEL, 1);
    if (!buffer)
    {
        child->status = ProgramStatus::DEAD;
        return false;
    }

    int childPageDirPaddr = memoryManager.vaddr2paddr(child->pageDirectoryAddress);
    int parentPageDirPaddr = memoryManager.vaddr2paddr(parent->pageDirectoryAddress);
    int *childPageDir = (int *)child->pageDirectoryAddress;
    int *parentPageDir = (int *)parent->pageDirectoryAddress;

    memset((void *)child->pageDirectoryAddress, 0, 768 * 4);

    for (int i = 0; i < 768; ++i)
    {
        if (!(parentPageDir[i] & 0x1))
        {
            continue;
        }

        int paddr = memoryManager.allocatePhysicalPages(AddressPoolType::USER, 1);
        if (!paddr)
        {
            child->status = ProgramStatus::DEAD;
            return false;
        }

        int pde = parentPageDir[i];
        int *pageTableVaddr = (int *)(0xffc00000 + (i << 12));

        asm_update_cr3(childPageDirPaddr);
        childPageDir[i] = (pde & 0x00000fff) | paddr;
        memset(pageTableVaddr, 0, PAGE_SIZE);
        asm_update_cr3(parentPageDirPaddr);
    }

    for (int i = 0; i < 768; ++i)
    {
        if (!(parentPageDir[i] & 0x1))
        {
            continue;
        }

        int *pageTableVaddr = (int *)(0xffc00000 + (i << 12));
        for (int j = 0; j < 1024; ++j)
        {
            if (!(pageTableVaddr[j] & 0x1))
            {
                continue;
            }

            int paddr = memoryManager.allocatePhysicalPages(AddressPoolType::USER, 1);
            if (!paddr)
            {
                child->status = ProgramStatus::DEAD;
                return false;
            }

            void *pageVaddr = (void *)((i << 22) + (j << 12));
            int pte = pageTableVaddr[j];

            // 父进程页 -> 内核缓冲区
            memcpy(buffer, pageVaddr, PAGE_SIZE);

            asm_update_cr3(childPageDirPaddr);
            pageTableVaddr[j] = (pte & 0x00000fff) | paddr;
            // 内核缓冲区 -> 子进程页
            memcpy(pageVaddr, buffer, PAGE_SIZE);
            asm_update_cr3(parentPageDirPaddr);
        }
    }

    memoryManager.releasePages(AddressPoolType::KERNEL, (int)buffer, 1);
    return true;
}
```

| 观察点 | ESP | 栈中关键内容 | 下一步跳转位置 |
| --- | --- | --- | --- |
| 进入 `asm_switch_thread` 前 | 待补充 | 当前父进程或内核线程调度栈 | `asm_switch_thread` |
| `mov esp, [next]` 后 | 待补充 | 子进程为首次运行伪造的线程切换栈 | `ret` |
| `ret` 后 | 待补充 | `asm_start_process` 参数为子进程 `ProcessStartStack` | `asm_start_process` |
| `asm_start_process` 中 | 待补充 | 子进程复制来的中断返回现场，`eax` 已改为 0 | `iret` |
| `iret` 后 | 待补充 | 用户栈恢复，回到 `fork` 调用之后 | 用户态父子分支判断处 |

父进程和子进程返回路径的相同点在于：最终都通过系统调用返回路径回到用户态，并且执行位置都在 `fork` 调用之后。

不同点在于：父进程是正在执行系统调用的进程，它从 `ProgramManager::fork` 返回后继续经过 `asm_system_call_handler` 和 `iret` 回到用户态；子进程则是之后才被调度器选中，它先经过 `asm_switch_thread` 切换到伪造的内核栈，再由 `asm_start_process` 使用复制来的 `ProcessStartStack` 返回用户态。

### A3 Wait/Exit 与进程生命周期

#### A3.1 Exit 的实现与分析

`exit` 用于结束当前进程。用户态程序调用 `exit(ret)` 后，会触发系统调用进入内核，最终执行 `ProgramManager::exit(ret)`。

测试代码如下。子进程显式调用 `exit(42)`，父进程暂不回收子进程，只用于观察子进程退出流程。

```c++
void first_process()
{
    int ret = fork();

    if (ret < 0)
    {
        printf("fork failed\n");
        asm_halt();
    }

    if (ret == 0)
    {
        printf("child: pid=%d, exit value=42\n", programManager.running->pid);
        exit(42);
    }
    else
    {
        printf("father: pid=%d, child pid=%d\n",
               programManager.running->pid, ret);
        asm_halt();
    }
}
```

> **截图位置 A3-1：子进程显式调用 `exit(42)` 的运行结果**  
> 建议文件：`./gallery/a3_exit_result.png`

`exit` 的执行过程主要包括保存退出值、释放资源、改变进程状态和触发调度。

首先，内核会关闭中断，避免退出过程被打断。然后取得当前运行进程的 PCB，把传入的退出值保存到 `retValue` 中，并把进程状态设置为 `DEAD`。退出值不会立即丢弃，因为父进程后续调用 `wait` 时还需要读取它。

随后，内核释放进程占用的用户空间资源。由于用户进程拥有独立页目录表，`exit` 需要遍历用户空间对应的页目录项和页表项，释放每个已映射用户虚拟页背后的用户物理页；页表本身占用的物理页也需要释放。用户页释放完成后，再释放进程页目录表占用的内核页，以及用户虚拟地址池位图占用的内核页。

进程状态变为 `DEAD` 后，`exit` 最后调用 `schedule`。调度器不会再把 `DEAD` 状态的进程重新放入就绪队列，因此该进程不会继续获得 CPU。对于用户进程，其 PCB 通常不会在 `exit` 中立刻释放，而是保留到父进程调用 `wait` 时回收。这样父进程还能取得子进程 pid 和退出值。

`ProgramManager::exit` 的关键代码如下。

```c++
void ProgramManager::exit(int ret)
{
    interruptManager.disableInterrupt();

    PCB *program = this->running;
    program->retValue = ret;
    program->status = ProgramStatus::DEAD;

    int *pageDir;
    int *page;
    int paddr;

    if (program->pageDirectoryAddress)
    {
        pageDir = (int *)program->pageDirectoryAddress;

        for (int i = 0; i < 768; ++i)
        {
            if (!(pageDir[i] & 0x1))
            {
                continue;
            }

            page = (int *)(0xffc00000 + (i << 12));

            for (int j = 0; j < 1024; ++j)
            {
                if (!(page[j] & 0x1))
                {
                    continue;
                }

                paddr = memoryManager.vaddr2paddr((i << 22) + (j << 12));
                memoryManager.releasePhysicalPages(AddressPoolType::USER, paddr, 1);
            }

            paddr = memoryManager.vaddr2paddr((int)page);
            memoryManager.releasePhysicalPages(AddressPoolType::USER, paddr, 1);
        }

        memoryManager.releasePages(AddressPoolType::KERNEL, (int)pageDir, 1);

        int bitmapBytes = ceil(program->userVirtual.resources.length, 8);
        int bitmapPages = ceil(bitmapBytes, PAGE_SIZE);
        memoryManager.releasePages(AddressPoolType::KERNEL,
                                   (int)program->userVirtual.resources.bitmap,
                                   bitmapPages);
    }

    schedule();
}
```

#### 正常返回为什么会隐式调用 `exit`

用户进程入口函数如果正常执行到函数结尾，也需要进入退出流程，否则函数返回后将不知道跳到哪里继续执行。

因此，在构造用户栈时，内核会在用户栈上预先放置一个返回地址，使用户进程入口函数返回时会跳转到 `exit`。同时栈上还会准备默认退出参数。这样即使用户函数没有显式调用 `exit`，函数正常返回后也会隐式进入 `exit` 系统调用，完成进程资源释放和状态切换。

从栈的角度看，用户进程刚进入入口函数时，用户栈已经被布置成“入口函数返回地址为 `exit`”的形态。入口函数末尾的 `ret` 指令会弹出这个地址，于是控制流转入 `exit`。

#### A3.2 Wait 的实现与父子进程同步

`wait` 用于父进程等待并回收已经退出的子进程。它既负责取得子进程退出值，也负责释放子进程 PCB，避免子进程退出后长期占用内核 PCB 资源。

测试代码如下。父进程连续创建两个子进程，两个子进程分别以不同返回值退出，父进程循环调用 `wait` 回收所有子进程。

```c++
void first_process()
{
    int pid = fork();
    if (pid == 0)
    {
        printf("child A: pid=%d, exit value=11\n", programManager.running->pid);
        exit(11);
    }

    pid = fork();
    if (pid == 0)
    {
        printf("child B: pid=%d, exit value=22\n", programManager.running->pid);
        exit(22);
    }

    int retval;
    while ((pid = wait(&retval)) != -1)
    {
        printf("father: collect child pid=%d, retval=%d\n", pid, retval);
    }

    printf("father: all child processes exit\n");
    asm_halt();
}
```

> **截图位置 A3-2：父进程回收多个子进程退出状态**  
> 建议文件：`./gallery/a3_wait_result.png`

`wait` 的执行逻辑如下。

父进程调用 `wait(&retval)` 后进入内核，内核遍历 `allPrograms` 队列，查找 `parentPid` 等于当前进程 pid 的 PCB。遍历时可能出现三种情况。

第一种情况是找到了属于当前父进程且状态为 `DEAD` 的子进程。此时说明该子进程已经退出，但 PCB 尚未回收。内核把子进程的 `retValue` 写入用户传入的 `retval` 指针，保存子进程 pid，然后调用 `releasePCB` 释放子进程 PCB，最后返回该子进程 pid。

第二种情况是当前进程已经没有任何子进程。此时 `wait` 没有可等待对象，直接返回 `-1`。父进程通常可以据此结束循环。

第三种情况是存在子进程，但还没有子进程进入 `DEAD` 状态。此时父进程需要等待。当前实现可以通过调用 `schedule` 主动让出 CPU，让子进程继续运行直到退出；如果进一步实现严格阻塞语义，也可以把父进程状态设置为 `BLOCKED`，并在子进程 `exit` 时唤醒父进程。

通过 `wait`，父进程和子进程之间形成了简单同步关系：子进程先执行并调用 `exit` 保存退出值，父进程随后通过 `wait` 收集退出值并释放子进程 PCB。

`ProgramManager::wait` 的核心代码如下。

```c++
int ProgramManager::wait(int *retval)
{
    PCB *child;
    ListItem *item;
    bool interrupt;
    bool hasChild;

    while (true)
    {
        interrupt = interruptManager.getInterruptStatus();
        interruptManager.disableInterrupt();

        item = this->allPrograms.head.next;
        hasChild = false;

        while (item)
        {
            child = ListItem2PCB(item, tagInAllList);

            if (child->parentPid == this->running->pid)
            {
                hasChild = true;
                if (child->status == ProgramStatus::DEAD)
                {
                    break;
                }
            }

            item = item->next;
        }

        if (item)
        {
            if (retval)
            {
                *retval = child->retValue;
            }

            int pid = child->pid;
            releasePCB(child);
            interruptManager.setInterruptStatus(interrupt);
            return pid;
        }

        if (!hasChild)
        {
            interruptManager.setInterruptStatus(interrupt);
            return -1;
        }

        interruptManager.setInterruptStatus(interrupt);
        schedule();
    }
}
```

#### A3.3 僵尸进程与孤儿进程处理

当前简化实现中存在两个典型问题。

第一个问题是僵尸进程。如果子进程已经调用 `exit`，状态变为 `DEAD`，但父进程一直不调用 `wait`，那么子进程 PCB 会一直保留在 `allPrograms` 队列中，无法被释放。这个子进程已经不能运行，但仍占用内核 PCB 资源，因此称为僵尸进程。

第二个问题是孤儿进程。如果父进程先于子进程退出，那么仍在运行的子进程会失去父进程。等这些子进程之后退出时，如果没有其他进程负责 `wait`，它们也可能变成无人回收的僵尸进程。

本实验计划采用方案 A：实现 `init` 进程托管孤儿进程，并由 `init` 负责回收。

具体思路如下：

- 系统创建第一个用户进程作为 `init` 进程，记录其 pid。
- 每个进程 PCB 中通过 `parentPid` 记录父进程 pid。
- 当某个父进程退出时，内核遍历 `allPrograms`，把所有 `parentPid` 等于该父进程 pid 的仍存活子进程改挂到 `init` 进程名下。
- `init` 进程循环调用 `wait`，持续回收已经退出的子进程。
- 如果普通父进程没有调用 `wait` 就退出，子进程会先变成孤儿进程，再由 `init` 进程托管并最终回收。

这种方案对应 Unix/Linux 中常见的 reparent 思路。它不会要求每个普通父进程都正确调用 `wait`，而是为系统提供一个兜底回收者，避免退出子进程长期滞留在 PCB 表中。

方案 A 的关键在于让系统中长期存在一个 `init` 进程，并在普通父进程退出时，把它的子进程重新挂到 `init` 名下。

首先在 `ProgramManager` 中增加 `initPid` 和托管子进程的辅助函数。

```c++

{
public:

    List readyPrograms;
    PCB *running;
    int USER_CODE_SELECTOR;
    int USER_DATA_SELECTOR;
    int USER_STACK_SELECTOR;

public:

};
```

在 `ProgramManager::initialize` 中初始化 `initPid`。

```c++

{
    allPrograms.initialize();
    readyPrograms.initialize();

    for (int i = 0; i < MAX_PROGRAM_AMOUNT; ++i)

        PCB_SET_STATUS[i] = false;
    }
```

当一个普通进程退出时，遍历所有 PCB，把它的子进程改挂到 `init`。这里不区分子进程是否已经 `DEAD`，因为已经退出但尚未回收的子进程也应该交给 `init` 调用 `wait` 回收。

```c++
void ProgramManager::adoptChildren(PCB *program)
{
    if (program->pid == initPid)
    {
        return;
    }

    ListItem *item = allPrograms.head.next;
    while (item)
    {
        PCB *child = ListItem2PCB(item, tagInAllList);
        if (child->parentPid == program->pid)
        {
            child->parentPid = initPid;
        }
        item = item->next;
    }
}
```

在 `exit` 中释放当前进程资源前后均可执行托管逻辑。更直观的写法是在标记当前进程为 `DEAD` 后，调用 `adoptChildren`。

```c++
void ProgramManager::exit(int ret)
{
    interruptManager.disableInterrupt();

    PCB *program = this->running;
    program->retValue = ret;
    program->status = ProgramStatus::DEAD;

    if (program->pageDirectoryAddress)
    {
        adoptChildren(program);
    }

    // 接下来仍然执行用户物理页、页表、页目录表和用户虚拟地址池 bitmap 的释放

    schedule();
}
```

`init` 进程本身循环调用 `wait`。当暂时没有子进程可回收时，`wait` 返回 `-1`，`init` 不退出，而是继续循环等待后续被托管的孤儿进程。

```c++
        }

}
```

```c++
void first_thread(void *arg)
{
    printf("start init process\n");

    int initPid = programManager.executeProcess((const char *)init_process, 1);
    programManager.initPid = initPid;

    printf("start orphan test process\n");
    programManager.executeProcess((const char *)orphan_test_process, 1);

    PCB *testProcess = ListItem2PCB(programManager.allPrograms.back(), tagInAllList);
    testProcess->parentPid = programManager.initPid;

    asm_halt();
}
```

这种实现后，普通父进程退出不会导致仍存活子进程无人管理；子进程退出后即使原父进程已经不存在，也会被 `init` 进程通过 `wait` 回收。

### 实验总结

本次实验把前面已经实现的线程调度、分页机制和同步基础进一步连接到用户进程模型中。

系统调用部分让我理解了用户态和内核态并不是简单的函数调用关系，而是由 CPU 特权级检查、IDT 中断门、TSS 内核栈、寄存器传参和 `iret` 返回共同构成的一套受控切换机制。尤其是 TSS 中 `esp0` 和 `ss0` 的作用，使每个用户进程在进入内核时能够使用独立内核栈。

进程创建部分的关键在于把“可调度的执行流”和“独立地址空间”结合起来。线程切换只需要保存和恢复栈，而进程切换还必须切换 `CR3`，并保证内核高地址空间在所有进程中保持一致。`ProcessStartStack` 则利用 `iret` 的返回规则，实现了从内核态启动用户态进程。

`fork` 展示了操作系统如何复制一个正在运行的进程。它不仅要复制用户空间和页表，还要构造子进程第一次调度时的内核栈，使子进程看起来像是从同一个系统调用返回。父子进程返回值不同的设计，则通过修改子进程保存现场中的 `eax` 实现。

`exit` 和 `wait` 构成了进程生命周期的闭环。进程退出时释放用户空间资源并保存退出状态，父进程通过 `wait` 回收 PCB 并取得返回值。僵尸进程和孤儿进程问题说明，进程退出不仅是单个进程自己的事情，还需要系统层面的父子关系管理和兜底回收机制。

整体来看，Lab8 将“特权级、系统调用、地址空间、调度、父子进程关系、资源回收”这些原本分散的概念串联到了一条完整路径中：用户程序通过系统调用进入内核，内核创建、调度、复制和终止进程，并最终保证进程资源能够被正确回收。
