##### `waitReady()` 等待磁盘就绪

ATA 主通道的状态寄存器位于端口 `0x1F7`。`DiskDriver::waitReady()` 通过循环读取该端口，等待状态寄存器中的 `BSY` 位清零：

```cpp
void DiskDriver::waitReady()
{
    uint8 status;
    do
    {
        asm_in_port(ATA_STATUS, &status);
    } while (status & ATA_STATUS_BSY);
}
```

`ATA_STATUS_BSY` 的值为 `0x80`。当该位为 1 时，表示磁盘控制器正在处理上一条命令，此时不应继续写入 LBA 地址、扇区数或命令寄存器。只有当 `BSY` 清零后，驱动才能继续设置本次读写请求。

##### `selectSector()` 设置 LBA 地址和扇区数

`selectSector()` 负责选择要访问的逻辑扇区：

```cpp
void DiskDriver::selectSector(uint32 lba, uint8 count)
{
    waitReady();
    asm_out_port(ATA_SECT_CNT, count);
    asm_out_port(ATA_LBA_LOW, (uint8)(lba & 0xFF));
    asm_out_port(ATA_LBA_MID, (uint8)((lba >> 8) & 0xFF));
    asm_out_port(ATA_LBA_HIGH, (uint8)((lba >> 16) & 0xFF));
    asm_out_port(ATA_DRIVE_HEAD, (uint8)(0xE0 | ((lba >> 24) & 0x0F)));
}
```

它首先调用 `waitReady()` 确认磁盘空闲，然后依次写入：

- `0x1F2`：扇区数量。
- `0x1F3`：LBA 地址低 8 位。
- `0x1F4`：LBA 地址中间 8 位。
- `0x1F5`：LBA 地址高 8 位。
- `0x1F6`：驱动器/磁头寄存器，`0xE0` 表示主盘和 LBA 模式，低 4 位保存 LBA 的 `[27:24]`。

本实验使用 LBA 模式，因此磁盘扇区可以被看成从 0 开始连续编号的逻辑块，驱动不需要处理传统 CHS 模式中的柱面、磁头和扇区换算。

##### `readSectors()` 读取流程

读取扇区时，驱动逐个扇区处理：

```cpp
bool DiskDriver::readSectors(uint32 lba, uint32 count, void *buffer)
{
    uint16 *buf = (uint16 *)buffer;

    for (uint32 s = 0; s < count; ++s)
    {
        selectSector(lba + s, 1);
        asm_out_port(ATA_COMMAND, ATA_CMD_READ);

        uint8 status;
        do
        {
            asm_in_port(ATA_STATUS, &status);
        } while (!(status & ATA_STATUS_DRQ));

        for (int i = 0; i < 256; ++i)
        {
            asm_in_port_word(ATA_DATA, &buf[s * 256 + i]);
        }
    }

    return true;
}
```

完整过程可以概括为：

1. 调用 `selectSector(lba + s, 1)` 选择当前要读取的扇区。
2. 向命令寄存器 `0x1F7` 写入 `ATA_CMD_READ`，即 `0x20`。
3. 轮询状态寄存器，等待 `DRQ` 位变为 1。`DRQ` 表示磁盘已经准备好数据，可以开始传输。
4. 从数据端口 `0x1F0` 连续执行 256 次 16 位输入，每次读取 2 字节，共读取 512 字节，也就是一个扇区。

##### `writeSectors()` 写入流程

写入流程与读取流程结构相似，但数据方向相反：

```cpp
bool DiskDriver::writeSectors(uint32 lba, uint32 count, const void *buffer)
{
    const uint16 *buf = (const uint16 *)buffer;

    for (uint32 s = 0; s < count; ++s)
    {
        selectSector(lba + s, 1);
        asm_out_port(ATA_COMMAND, ATA_CMD_WRITE);

        uint8 status;
        do
        {
            asm_in_port(ATA_STATUS, &status);
        } while (!(status & ATA_STATUS_DRQ));

        for (int i = 0; i < 256; ++i)
        {
            asm_out_port_word(ATA_DATA, buf[s * 256 + i]);
        }
    }

    return true;
}
```

两者的相同点是都需要先设置 LBA 地址、扇区数，并等待 `DRQ` 置位后再通过数据端口传输一个扇区的数据。不同点在于读取发送 `0x20` 命令并使用 `asm_in_port_word` 从端口读入数据；写入发送 `0x30` 命令并使用 `asm_out_port_word` 将缓冲区数据写到端口。

##### 为什么数据寄存器使用 16 位 I/O

ATA 数据端口 `0x1F0` 按规范是 16 位数据端口。一个扇区大小为 512 字节，也就是 256 个 16 位字，因此本实验使用：

```asm
in ax, dx
out dx, ax
```

而不是：

```asm
in al, dx
out dx, al
```

如果只使用 8 位 I/O，每次只能传输一个字节，不符合 ATA 数据寄存器的传输宽度约定，也会破坏驱动对“每扇区 256 个 word”的处理逻辑。使用 16 位 I/O 可以一次完成一个 ATA 数据字的传输，正好对应硬盘控制器的数据通路和扇区大小。

#### A1.2 FAT16 文件系统布局分析

本实验的文件系统从磁盘第 500 个扇区开始，避开 MBR、Bootloader 和 Kernel 所在区域。结合 `src/1/include/fat16.h` 中的常量，布局如下：

| 磁盘扇区范围 | 内容 | 大小 |
| --- | --- | --- |
| 0 | MBR | 1 扇区 |
| 1 - 5 | Bootloader | 5 扇区 |
| 6 - 499 | Kernel 与预留区域 | 为内核和对齐预留 |
| 500 | SuperBlock | 1 扇区 |
| 501 - 532 | FAT 表 | 32 扇区 |
| 533 - 564 | 根目录区 | 32 扇区 |
| 565 起 | 数据区 | 每簇 1 扇区 |

对应关系也可以画成：

```text
0        1             6                         500       501           533           565
+--------+-------------+-------------------------+----------+-------------+-------------+----------------
|  MBR   | Bootloader  | Kernel / Reserved Gap   | SuperBlk | FAT Table   | Root Dir    | Data Clusters
+--------+-------------+-------------------------+----------+-------------+-------------+----------------
 1 sector   5 sectors         reserved before FS    1 sector   32 sectors    32 sectors   cluster 2 ...
```

其中：

```cpp
#define FS_START_SECTOR   500
#define FAT_SECTORS       32
#define ROOT_DIR_SECTORS  32
#define MAX_ROOT_ENTRIES  512
#define ENTRIES_PER_SECTOR 16
#define FAT_ENTRIES       8192
#define DATA_START_OFFSET 65
#define SECTOR_SIZE       512
```

`DATA_START_OFFSET = 1 + 32 + 32 = 65`，因此数据区起始扇区为 `500 + 65 = 565`。

FAT 表每个条目占 2 字节，即 16 bit。FAT 表占用 32 个扇区，每个扇区 512 字节，因此共有：

```text
32 * 512 / 2 = 8192
```

个 FAT 条目。其中 FAT[0] 和 FAT[1] 为保留项，实际可用于管理数据区的簇号从 2 到 8191，因此最多可管理 8190 个数据簇。由于本实验中每簇为 1 个扇区，所以最多可管理约 `8190 * 512` 字节的数据区空间。

根目录区每个目录条目占 32 字节，每个扇区可存放：

```text
512 / 32 = 16
```

个目录条目。根目录区共 32 个扇区，因此最多存放：

```text
32 * 16 = 512
```

个文件或目录条目。本实验只实现平面根目录，因此这个值也是当前文件系统最多支持的文件数。

如果一个文件占用了簇链：

```text
5 -> 8 -> 12 -> EOF
```

则 FAT 表中的值为：

| FAT 条目 | 值 | 含义 |
| --- | --- | --- |
| `FAT[5]` | `8` | 簇 5 后继是簇 8 |
| `FAT[8]` | `12` | 簇 8 后继是簇 12 |
| `FAT[12]` | `0xFFF8` | 簇 12 是文件末尾 |

`clusterToSector()` 的实现如下：

```cpp
uint32 FAT16::clusterToSector(uint16 cluster)
{
    return FS_START_SECTOR + DATA_START_OFFSET + (cluster - 2);
}
```

簇号从 2 开始，是 FAT 文件系统的传统约定。FAT[0] 和 FAT[1] 用作保留项，不对应实际数据簇。因此簇 2 对应数据区的第一个扇区，也就是扇区 565；簇 3 对应扇区 566，以此类推。公式中的 `(cluster - 2)` 正是把 FAT 簇号转换为数据区内部从 0 开始的偏移。

### A2 文件操作实现分析

#### A2.1 文件创建与写入

##### `createFile("hello.txt")` 执行流程

创建文件前，文件系统首先检查是否已经挂载：

```cpp
if (!mounted)
{
    printf("Error: filesystem not mounted\n");
    return false;
}
```

随后调用 `findEntry(name)` 检查根目录中是否已经存在同名文件。`findEntry` 内部会先调用 `toFAT16Name` 将普通文件名转换为 8.3 格式。对于 `"hello.txt"`，转换结果为：

```text
HELLO   TXT
```

其中 `HELLO` 存入 8 字节文件名字段，不足 8 字节的部分用空格补齐；`TXT` 存入 3 字节扩展名字段。转换过程中，小写字母会被改成大写，文件名超过 8 个字符或扩展名超过 3 个字符的部分会被截断。

如果没有找到重复文件，`createFile` 调用 `findFreeEntry()` 扫描根目录区。根目录中的条目有两类可以复用：

- 首字节为 `0x00`：空条目，并且后续条目也为空。
- 首字节为 `0xE5`：之前删除过的条目。

找到空闲位置后，函数初始化一个新的 `DirEntry`：

```cpp
DirEntry entry;
memset(&entry, 0, sizeof(DirEntry));

char fat16Name[11];
toFAT16Name(name, fat16Name);
memcpy(fat16Name, entry.filename, 8);
memcpy(fat16Name + 8, entry.extension, 3);

entry.attributes = ATTR_ARCHIVE;
entry.firstCluster = 0;
entry.fileSize = 0;
```

空文件暂时不占用数据簇，因此 `firstCluster = 0`，`fileSize = 0`。最后通过 `writeDirEntry(idx, &entry)` 写回根目录区。

##### `writeFile("hello.txt", data, size)` 执行流程

`writeFile` 采用覆盖写模式。函数首先查找目标文件：

- 如果文件不存在，则找到一个空闲目录条目并创建元数据。
- 如果文件已经存在，则读取原目录条目，并释放旧数据对应的簇链。

释放旧簇链的关键代码为：

```cpp
if (entry.firstCluster >= 2)
{
    freeClusterChain(entry.firstCluster);
}
```

`freeClusterChain` 从文件起始簇出发，沿着 FAT 表中的后继簇号逐个释放：

```cpp
void FAT16::freeClusterChain(uint16 startCluster)
{
    uint16 current = startCluster;
    while (current >= 2 && current < FAT16_BAD)
    {
        uint16 next = fatTable[current];
        fatTable[current] = FAT16_FREE;
        current = next;
    }
}
```

这样，覆盖写不会泄漏旧文件占用的簇。

新内容需要的簇数通过向上取整得到：

```cpp
uint32 clustersNeeded = (size + SECTOR_SIZE - 1) / SECTOR_SIZE;
```

由于每簇为 512 字节，若文件大小为 1 到 512 字节，需要 1 个簇；若为 513 到 1024 字节，需要 2 个簇。

每次分配簇时，`allocateCluster()` 从 FAT[2] 开始线性查找空闲项：

```cpp
uint16 FAT16::allocateCluster()
{
    for (int i = 2; i < FAT_ENTRIES; ++i)
    {
        if (fatTable[i] == FAT16_FREE)
        {
            fatTable[i] = FAT16_EOF;
            return (uint16)i;
        }
    }
    return 0;
}
```

新分配的簇会先被标记为 EOF。如果后续还需要继续分配簇，则将前一个簇的 FAT 表项改成当前簇号，从而构造出 FAT 链：

```cpp
if (i == 0)
    firstCluster = cluster;
else
    fatTable[prevCluster] = cluster;
```

文件数据被按 512 字节拆分写入，每个簇对应一个扇区：

```cpp
uint8 sectorBuf[SECTOR_SIZE];
memset(sectorBuf, 0, SECTOR_SIZE);
uint32 copySize = size - i * SECTOR_SIZE;
if (copySize > SECTOR_SIZE)
    copySize = SECTOR_SIZE;
memcpy((void *)(data + i * SECTOR_SIZE), sectorBuf, copySize);
disk->writeSectors(clusterToSector(cluster), 1, sectorBuf);
```

最后更新目录条目的起始簇号和文件大小，并同步 FAT 表：

```cpp
entry.firstCluster = firstCluster;
entry.fileSize = size;
writeDirEntry(idx, &entry);
syncFAT();
```

必须调用 `syncFAT()` 的原因是，本实验在内存中维护了一份 `fatTable` 缓存。分配、链接和释放簇时，首先改变的是内存中的 FAT 表；如果不写回磁盘，重启或重新挂载后磁盘上的 FAT 表仍然是旧状态，目录条目中的 `firstCluster` 就无法正确找到文件内容。

#### A2.2 文件读取与删除

##### `readFile("hello.txt", buffer, maxSize)` 执行流程

读取文件时，函数先通过 `findEntry` 找到对应目录条目。目录条目中最重要的两个字段是：

- `firstCluster`：文件内容所在簇链的起始簇号。
- `fileSize`：文件实际字节数。

读取大小取文件大小和调用者缓冲区大小中的较小者：

```cpp
uint32 readSize = entry.fileSize;
if (readSize > maxSize)
    readSize = maxSize;
```

随后从起始簇开始遍历 FAT 链：

```cpp
while (cluster >= 2 && cluster < FAT16_BAD && bytesRead < readSize)
{
    disk->readSectors(clusterToSector(cluster), 1, sectorBuf);

    uint32 copySize = readSize - bytesRead;
    if (copySize > SECTOR_SIZE)
        copySize = SECTOR_SIZE;

    memcpy(sectorBuf, buffer + bytesRead, copySize);
    bytesRead += copySize;

    cluster = fatTable[cluster];
}
```

每轮循环读取一个簇对应的扇区，将其中有效数据复制到用户缓冲区，然后根据 `fatTable[cluster]` 转到下一个簇。读取过程会在两种情况下停止：

- FAT 链到达 EOF 或无效簇号，即簇号不再满足 `cluster >= 2 && cluster < FAT16_BAD`。
- 已经读取到 `readSize` 字节，也就是达到文件大小或 `maxSize` 限制。

##### `deleteFile("data.bin")` 执行流程

删除文件首先查找目录项并读出 `DirEntry`。如果该文件占用簇，则调用 `freeClusterChain` 释放完整簇链，并通过 `syncFAT()` 将修改后的 FAT 表写回磁盘：

```cpp
if (entry.firstCluster >= 2)
{
    freeClusterChain(entry.firstCluster);
    syncFAT();
}
```

随后将目录条目的第一个字节设置为 `0xE5`：

```cpp
entry.filename[0] = (char)ATTR_DELETED;
writeDirEntry(idx, &entry);
```

这表示该目录项已经被删除，后续 `findFreeEntry()` 可以复用这个位置。

需要注意的是，删除文件并不会真正擦除磁盘数据区中的内容。它只是把 FAT 表中的对应簇标记为空闲，并把目录项标记为删除。只要这些簇还没有被新文件覆盖，原始数据仍然可能保留在磁盘镜像中，因此真实文件系统中删除文件往往存在数据恢复可能，也带来安全删除问题。

### A3 文件系统功能扩展

#### A3.1 文件追加写入

当前 `writeFile` 是覆盖写模式。追加写入需要在保留旧文件内容的基础上，将新数据写到文件末尾。接口设计为：

```cpp
bool FAT16::appendFile(const char *name, const char *data, uint32 size);
```

实现思路如下：

1. 如果 `size == 0`，直接返回成功。
2. 如果文件不存在，则调用 `writeFile(name, data, size)`，行为等同于新建并写入。
3. 如果文件存在，读取目录条目，得到 `firstCluster` 和旧 `fileSize`。
4. 沿 FAT 链找到最后一个簇。
5. 根据 `fileSize % SECTOR_SIZE` 判断最后一个簇是否还有剩余空间。
6. 如果最后一个簇未满，先读取该簇所在扇区，把能够放入的追加数据补到扇区末尾，再写回该扇区。
7. 如果还有剩余追加数据，则继续调用 `allocateCluster()` 分配新簇，并把旧链表尾部指向新簇。
8. 更新目录条目中的 `fileSize`，写回目录项并调用 `syncFAT()`。

关键伪代码如下：

```cpp
bool FAT16::appendFile(const char *name, const char *data, uint32 size)
{
    if (size == 0)
        return true;

    int idx = findEntry(name);
    if (idx < 0)
        return writeFile(name, data, size);

    DirEntry entry;
    readDirEntry(idx, &entry);

    if (entry.firstCluster < 2 || entry.fileSize == 0)
        return writeFile(name, data, size);

    uint16 last = entry.firstCluster;
    while (fatTable[last] >= 2 && fatTable[last] < FAT16_BAD)
        last = fatTable[last];

    uint32 written = 0;
    uint32 offset = entry.fileSize % SECTOR_SIZE;

    if (offset != 0)
    {
        uint8 sectorBuf[SECTOR_SIZE];
        disk->readSectors(clusterToSector(last), 1, sectorBuf);

        uint32 canWrite = SECTOR_SIZE - offset;
        if (canWrite > size)
            canWrite = size;

        memcpy((void *)data, sectorBuf + offset, canWrite);
        disk->writeSectors(clusterToSector(last), 1, sectorBuf);
        written += canWrite;
    }

    while (written < size)
    {
        uint16 cluster = allocateCluster();
        if (cluster == 0)
            return false;

        fatTable[last] = cluster;
        last = cluster;

        uint8 sectorBuf[SECTOR_SIZE];
        memset(sectorBuf, 0, SECTOR_SIZE);

        uint32 copySize = size - written;
        if (copySize > SECTOR_SIZE)
            copySize = SECTOR_SIZE;

        memcpy((void *)(data + written), sectorBuf, copySize);
        disk->writeSectors(clusterToSector(cluster), 1, sectorBuf);
        written += copySize;
    }

    entry.fileSize += size;
    writeDirEntry(idx, &entry);
    syncFAT();
    return true;
}
```

测试方式是在 `runDemo()` 中对同一个文件多次追加：


#### A3.2 磁盘空间统计与碎片率

文件占用簇数统计接口如下：

```cpp
int FAT16::getFileClusterCount(const char *name);
```

实现时先找到目录条目。如果文件不存在或为空文件，返回 0；否则从 `firstCluster` 开始沿 FAT 链遍历，每经过一个有效簇就将计数加一：

```cpp
int FAT16::getFileClusterCount(const char *name)
{
    int idx = findEntry(name);
    if (idx < 0)
        return 0;

    DirEntry entry;
    readDirEntry(idx, &entry);

    int count = 0;
    uint16 cluster = entry.firstCluster;
    while (cluster >= 2 && cluster < FAT16_BAD)
    {
        ++count;
        cluster = fatTable[cluster];
    }

    return count;
}
```

碎片率接口如下：

```cpp
float FAT16::getFragmentation();
```

本实验可将碎片率定义为：

```text
碎片率 = 不连续簇链接次数 / 总簇链接次数
```

也就是遍历所有文件的簇链时，如果某个簇的后继簇不是 `当前簇 + 1`，则认为发生了一次不连续分配。单簇文件没有簇间链接，不计入分母。

参考实现思路如下：

```cpp
float FAT16::getFragmentation()
{
    int totalLinks = 0;
    int fragmentedLinks = 0;

    DirEntry entry;
    for (int i = 0; i < MAX_ROOT_ENTRIES; ++i)
    {
        readDirEntry(i, &entry);
        if ((uint8)entry.filename[0] == 0x00)
            break;
        if ((uint8)entry.filename[0] == ATTR_DELETED)
            continue;

        uint16 current = entry.firstCluster;
        while (current >= 2 && current < FAT16_BAD)
        {
            uint16 next = fatTable[current];
            if (next >= 2 && next < FAT16_BAD)
            {
                ++totalLinks;
                if (next != current + 1)
                    ++fragmentedLinks;
            }
            current = next;
        }
    }

    if (totalLinks == 0)
        return 0.0f;
    return (float)fragmentedLinks / (float)totalLinks;
}
```

测试时可以先连续创建多个文件，此时 FAT 分配通常较连续，碎片率接近 0；再删除中间文件制造空洞，然后创建更大的文件，让它复用空洞并继续在后方分配新簇，碎片率会升高。

示例测试流程如下：

```cpp
fs->writeFile("a.txt", bigA, strlen(bigA));
fs->writeFile("b.txt", bigB, strlen(bigB));
fs->writeFile("c.txt", bigC, strlen(bigC));
printf("fragmentation before delete: %d%%\n", (int)(fs->getFragmentation() * 100));

fs->deleteFile("b.txt");
fs->writeFile("d.txt", largerD, strlen(largerD));
printf("fragmentation after reuse: %d%%\n", (int)(fs->getFragmentation() * 100));
printf("d.txt clusters: %d\n", fs->getFileClusterCount("d.txt"));
```

#### A3.3 文件重命名

文件重命名接口如下：

```cpp
bool FAT16::renameFile(const char *oldName, const char *newName);
```

重命名只改变目录条目中的文件名字段，不需要移动文件数据，也不需要修改 FAT 链。因此它的关键是验证旧文件存在、新文件名不冲突，然后重写 `filename` 和 `extension` 字段。

参考实现思路如下：

```cpp
bool FAT16::renameFile(const char *oldName, const char *newName)
{
    int oldIdx = findEntry(oldName);
    if (oldIdx < 0)
        return false;

    if (findEntry(newName) >= 0)
        return false;

    DirEntry entry;
    readDirEntry(oldIdx, &entry);

    char fat16Name[11];
    toFAT16Name(newName, fat16Name);
    memcpy(fat16Name, entry.filename, 8);
    memcpy(fat16Name + 8, entry.extension, 3);

    writeDirEntry(oldIdx, &entry);
    return true;
}
```

测试流程如下：

```cpp
fs->writeFile("old.txt", "rename test", strlen("rename test"));
fs->renameFile("old.txt", "new.txt");

readBuf[n] = '\0';

```

预期现象是目录中不再出现 `old.txt`，而出现 `new.txt`；读取 `new.txt` 得到的内容仍然是原文件内容。这说明重命名只改变目录项元数据，没有影响文件数据所在簇链。

### 思考题

1. 在真实的 FAT16 文件系统中，BPB 存储在哪里？它与我们的 SuperBlock 有什么区别？

真实 FAT16 的 BPB 位于卷的引导扇区中，也就是 FAT 分区的第一个扇区。该扇区除了 BPB，还包含跳转指令、OEM 名称、扩展 BPB、引导代码和结束标志 `0x55AA` 等内容。BPB 会记录每扇区字节数、每簇扇区数、保留扇区数、FAT 表数量、根目录项数量、总扇区数、介质描述符和每个 FAT 占用扇区数等信息。

本实验的 `SuperBlock` 是简化版结构，只保存 `"SF16"` 魔数、扇区大小、簇大小、FAT 表大小、根目录区大小和最大根目录项数等核心信息。它不需要兼容真实 BIOS 启动流程，也没有 FAT 表副本数量、隐藏扇区、卷标等完整 FAT16 元数据。

2. FAT16 的最大卷大小限制是多少？这个限制来源于哪些因素？FAT32 如何突破这些限制？

FAT16 的核心限制来自 FAT 表条目宽度。每个 FAT16 条目只有 16 位，可表示的簇数量有限，实际可用簇数通常小于 65536。卷大小大致等于“可用簇数 × 每簇大小”。在常见兼容设置中，FAT16 最大卷大小通常约为 2GB；部分系统使用 64KB 簇时可以支持更大容量，但兼容性较差。

FAT32 将 FAT 表条目扩展到 32 位，其中通常使用低 28 位表示簇号，因此可以管理远多于 FAT16 的簇。同时 FAT32 的根目录不再固定在 FAT 表之后的一个定长区域，而是像普通文件一样由簇链管理，这也突破了 FAT16 固定根目录大小的限制。

3. PIO 模式下 CPU 必须全程参与数据传输。DMA 模式如何改进这一点？

PIO 模式中，CPU 需要不断轮询状态寄存器，并通过 `in` / `out` 指令逐字搬运数据。每读取或写入一个扇区，CPU 都要参与 256 次 16 位端口 I/O。

DMA 模式下，操作系统先设置好内存缓冲区地址、传输长度和方向，然后由磁盘控制器或总线主控设备直接在磁盘和内存之间搬运数据。CPU 不再负责逐字复制，只需要在传输开始时配置 DMA，并在中断通知完成后处理结果。这样可以降低 CPU 占用，提高大块数据读写性能。

4. 当前实现将整个 FAT 表缓存在内核内存中。如果 FAT 表非常大，这种策略还可行吗？如何改进？

对于本实验的 8192 项 FAT 表，整体缓存非常简单且开销很小。但如果换成大型 FAT32 卷，FAT 表可能占用大量空间，全部读入内存会浪费内存，也会让挂载和同步成本变高。

改进方法包括：只按需缓存 FAT 表的一部分扇区；为 FAT 扇区建立块缓存和脏位；采用 LRU 等策略淘汰长期不用的 FAT 块；修改后延迟写回或批量写回；在一致性要求更高的系统中，还可以配合日志或事务机制降低崩溃后的不一致风险。

5. 文件系统日志机制如何防止意外断电导致的数据不一致？Ext4 与 FAT 有何不同？

日志机制的核心思想是在真正修改文件系统元数据之前，先把即将执行的修改作为事务写入日志区域。若系统在中途断电，重启后文件系统可以根据日志判断某个事务是完整提交还是未完成，从而进行重放或回滚，避免出现目录项已经更新但 FAT 或位图没有更新这类不一致状态。

Ext4 支持日志机制，通常至少会对元数据修改进行日志保护，也可以根据挂载模式处理数据日志。FAT 文件系统本身没有通用日志机制，崩溃后通常依赖 fsck 或 scandisk 扫描 FAT 表和目录项来修复错误，因此面对突然断电时的一致性保证更弱。

6. 使用 `0xE5` 标记删除文件但不擦除数据，对数据恢复和数据安全意味着什么？

这意味着删除操作更多是“解除引用”，不是“清除内容”。目录项被标记为删除，FAT 链被释放后，原文件数据仍可能保留在原来的数据簇中，直到这些簇被新文件覆盖。因此数据恢复工具可以根据残留目录项和数据区内容尝试恢复已删除文件。

从安全角度看，这会导致敏感数据泄露风险。安全删除需要主动覆盖文件占用的数据簇，清理目录项中的元数据，并考虑现代 SSD 的磨损均衡和 TRIM 机制。另一种更可靠的方式是使用全盘或文件级加密，删除时销毁密钥，使残留密文失去可解密性。

### 实验总结与心得体会

本次实验把磁盘驱动和文件系统两个层次连接了起来。ATA PIO 驱动让我理解了操作系统如何通过端口 I/O 直接控制硬件：先等待磁盘不忙，再设置 LBA 地址和扇区数，发送读写命令，最后通过 16 位数据端口搬运一个扇区的数据。虽然 PIO 方式效率不高，但它清楚展示了块设备驱动的基本控制流程。

FAT16 文件系统部分让我认识到，文件系统的核心并不只是“把数据写到磁盘”，而是要同时维护空间分配信息、目录元数据和文件内容之间的一致关系。目录项记录文件从哪里开始，FAT 表记录文件后续簇链，数据区保存实际内容。任何一次写入、覆盖或删除操作，都必须让这三者保持一致。

在扩展功能设计中，追加写入要求正确处理最后一个未满簇和新簇分配；碎片率统计要求遍历文件簇链并识别不连续链接；重命名则说明有些文件操作只需要修改元数据，而不需要移动数据本身。通过这些功能，可以更直观地理解 FAT 链式分配结构的优点和局限：实现简单、元数据直观，但随机访问和碎片管理能力较弱。

总体来看，本实验让我从“磁盘扇区”这一低层抽象逐步走到“文件”这一用户可见抽象，也加深了我对操作系统存储子系统分层设计的理解。
