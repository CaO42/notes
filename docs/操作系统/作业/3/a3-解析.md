## 一、选择题

1. C

解析：进程的典型内存布局包括代码段、数据段、堆和栈。页表是操作系统和硬件用于地址转换的数据结构，不属于进程用户地址空间中的典型段。

2. C

解析：程序是存放在磁盘上的被动实体，只有被装入内存并开始执行后才成为进程。进程是动态的、主动的执行实体。同一个程序可以同时对应多个进程。

3. C

解析：等待态进程等待的事件发生后，通常先进入就绪态，等待 CPU 调度；不能直接从等待态变为运行态。

4. B

解析：长期调度器决定哪些作业进入内存成为可运行进程，因此控制多道程序的度，执行频率较低。短期调度器从就绪队列中选择下一个占用 CPU 的进程，执行频率很高。

5. C

解析：子进程已经终止，但父进程尚未调用 `wait()` 回收其退出状态时，子进程仍保留少量进程表项，因此处于僵尸态。

6. B

解析：消息传递模型通过发送和接收消息进行通信，不要求通信进程共享地址空间，因此更适合分布式系统。共享内存建立后通常不需要每次通信都让内核介入。

7. B

解析：当 `send()` 和 `receive()` 都采用阻塞方式时，发送方和接收方需要在通信点相互等待，这种方式称为会合。

8. C

解析：`fork()` 成功后会返回两次：在父进程中返回子进程 PID，在子进程中返回 0。如果失败，才返回 -1。

9. C

解析：PCB 保存进程状态、程序计数器、CPU 寄存器、调度信息、内存管理信息、打开文件列表等。源代码文本不是 PCB 的典型内容。

10. C

解析：浏览器为不同标签页创建独立渲染进程，可以增强隔离性。某个标签页崩溃时，通常不会直接导致其他标签页或整个浏览器崩溃。

## 二、简答题

### 1. fork 程序的所有可能输出

程序输出为：

```text
child: x = 15
parent: x = 2
```

理由如下：

1. `fork()` 后产生父进程和子进程，它们拥有各自独立的地址空间。
2. `fork()` 前 `x = 5`。子进程中 `pid == 0`，执行 `x += 10`，所以子进程中的 `x` 变为 15。
3. 父进程中 `pid > 0`，执行 `x -= 3`，所以父进程中的 `x` 变为 2。
4. 父进程在打印前调用 `wait(NULL)`，会等待子进程结束。因此在正常 `fork()` 成功的情况下，子进程的输出会先出现，父进程的输出会后出现。

关键点是：父子进程中的 `x` 是各自地址空间里的副本，子进程对 `x` 的修改不会影响父进程。

### 2. 共享内存与消息传递的比较

共享内存的优点：

1. 速度快。共享区域建立后，进程可以直接读写同一块内存，不需要每次数据交换都复制到内核。
2. 适合大量数据交换，例如本机上的高速生产者-消费者模型、多媒体数据处理等。
3. 通信开销低，尤其适合高频率、低延迟场景。

共享内存的缺点：

1. 需要额外同步机制，如互斥锁、信号量、条件变量，否则容易出现竞态条件。
2. 编程复杂，错误更隐蔽。
3. 通常更适合同一台机器上的进程，不适合天然跨网络通信。

消息传递的优点：

1. 抽象简单，进程通过 `send()` 和 `receive()` 交换消息。
2. 内核或通信系统可以帮助完成同步、保护和排队。
3. 更适合分布式系统，因为通信双方不需要共享地址空间。

消息传递的缺点：

1. 通常需要内核介入，可能产生上下文切换和数据复制开销。
2. 传输大量数据时效率可能不如共享内存。
3. 消息队列容量、阻塞语义和通信链路管理会影响设计。

选择上，如果是同一主机中大量数据、高吞吐、低延迟通信，我倾向于共享内存。如果是控制消息、跨主机通信、强调隔离性和接口清晰的场景，我倾向于消息传递。

### 3. 僵尸进程与孤儿进程

僵尸进程是已经终止但其父进程还没有调用 `wait()` 或 `waitpid()` 回收退出状态的子进程。此时子进程不再执行代码，也不再占用完整运行资源，但系统仍保留它的 PID、退出状态等少量进程表信息，供父进程查询。

僵尸进程的产生过程是：子进程先于父进程结束，内核保存其退出信息；如果父进程长时间不回收，僵尸进程就会一直留在进程表中。处理方式是父进程调用 `wait()`/`waitpid()` 回收；如果父进程退出，僵尸子进程通常会被 `init` 或 `systemd` 收养并最终回收。

孤儿进程是父进程已经终止，但子进程仍在运行的进程。孤儿进程本身不是错误状态，它仍然可以继续执行。

孤儿进程的产生过程是：父进程先于子进程退出。操作系统会把孤儿进程重新指定给 `init` 或 `systemd` 等祖先进程收养；当孤儿进程以后终止时，由收养它的进程负责回收退出状态。

### 4. fork 循环产生的进程数

循环执行 3 次，每次 `fork()` 都会让当前已有的每个进程再复制出一个新进程，因此进程数每轮翻倍：

1. 初始时有 1 个进程。
2. 第 1 次 `fork()` 后有 2 个进程。
3. 第 2 次 `fork()` 后有 4 个进程。
4. 第 3 次 `fork()` 后有 8 个进程。

所以总共会产生 8 个进程，包括最初的进程。

循环结束后，每个进程都会继续执行 `printf("hello\n")`，因此 "LINE J" 处的 `printf` 会被执行 8 次，输出 8 行 `hello`。这些输出行之间的先后顺序由调度器决定，不固定。

## 三、分析题

### 1. 循环缓冲区为什么只能使用 BUFFER_SIZE - 1 项，以及改进方案

#### （a）原因

教材中的循环缓冲区通常使用数组 `buffer[BUFFER_SIZE]`，并用两个下标 `in` 和 `out` 管理：

```c
item buffer[BUFFER_SIZE];
int in = 0;
int out = 0;
```

其中，`in` 指向下一个要写入的位置，`out` 指向下一个要读出的位置。

这种方案一般约定：

```c
in == out
```

表示缓冲区为空。

如果允许使用全部 `BUFFER_SIZE` 个位置，那么当缓冲区被写满后，`in` 也会再次追上 `out`，此时同样有：

```c
in == out
```

这样就无法区分“空”和“满”。为了避免歧义，教材方案故意空出一个位置，把判满条件定义为：

```c
(in + 1) % BUFFER_SIZE == out
```

因此在 `BUFFER_SIZE = 10` 时，最多只能存放 9 个缓冲项。

#### （b）改进方案

可以增加一个计数变量 `count`，记录当前缓冲区中已有的元素个数。关键数据结构如下：

```c
#define BUFFER_SIZE 10

item buffer[BUFFER_SIZE];
int in = 0;
int out = 0;
int count = 0;
```

判空条件：

```c
count == 0
```

判满条件：

```c
count == BUFFER_SIZE
```

生产者插入元素时：

```c
buffer[in] = next_produced;
in = (in + 1) % BUFFER_SIZE;
count++;
```

消费者取出元素时：

```c
next_consumed = buffer[out];
out = (out + 1) % BUFFER_SIZE;
count--;
```

这样即使 `in == out`，也可以通过 `count` 判断当前是空还是满。当 `count == 0` 时为空；当 `count == BUFFER_SIZE` 时为满，因此所有 10 个缓冲项都可以被利用。

在并发实现中，`in`、`out` 和 `count` 的修改必须放在互斥保护下，例如使用 mutex；也可以配合两个信号量 `empty = BUFFER_SIZE` 和 `full = 0` 管理空槽和已有元素数量。
