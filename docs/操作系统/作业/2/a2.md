### 二

#### 1

面向用户便利性的服务包括：用户界面、程序执行、I/O 操作、文件系统操作、通信、错误检测等。它们主要关注用户和应用程序能否方便地使用计算机系统、完成具体任务。

面向系统高效运行的服务包括：资源分配、记账、保护与安全等。它们主要关注多个进程或用户共享系统时，系统能否公平、高效、可靠和安全地运行。

二者区别在于：前者更直接面向用户体验和程序功能，后者更侧重操作系统内部资源管理和整体运行质量。

#### 2

系统调用接口是应用程序请求内核服务的接口。应用程序通常调用 API，例如 POSIX API 或 Win32 API，而不是直接执行底层系统调用。

应用程序调用 API 后，运行时库会把 API 调用转换为具体系统调用，准备系统调用号和参数，然后通过陷入指令进入内核态。内核根据系统调用号执行对应服务，再把结果返回给运行时库和应用程序。

因此，API 是程序员使用的高级接口，system-call interface 是内核提供的实际入口，运行时库负责把二者连接起来。

#### 3

分层操作系统结构的优点是结构清晰、模块化程度高、便于调试、维护和验证。上层建立在下层服务之上，接口明确，适合逐步构造系统。

局限性是层次划分并不总是容易，某些功能可能跨层依赖；同时，请求逐层传递可能带来额外开销，严格分层也可能降低系统灵活性。

在教学操作系统中，分层设计可以帮助学生从硬件抽象到内核服务再到用户接口逐步理解系统结构，也便于把实验拆成相对独立的模块。

### 三

#### 1

如果把压缩功能直接编入内核，优点是调用路径短，可能减少上下文切换和数据复制，性能较好，适合文件系统透明压缩等与内核 I/O 路径紧密结合的场景。缺点是安全风险较高，一旦压缩代码出错可能导致系统崩溃或产生内核级漏洞；同时内核代码调试和升级更困难。

如果把压缩服务放到用户态，优点是隔离性好，服务崩溃通常不会影响整个系统；也更容易调试、替换算法和复用现有压缩库。缺点是需要通过系统调用与内核交互，可能增加上下文切换和数据复制开销。

如果是普通文件压缩工具、教学实验或需要频繁迭代的服务，我更倾向于选择用户态方案。如果是高性能透明文件系统压缩，且代码经过充分验证并必须位于 I/O 关键路径中，可以考虑内核态方案。
