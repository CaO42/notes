2. B

3. A

解析：系统调用是用户程序请求操作系统内核服务的编程接口。例如文件读写、创建进程、申请内存、访问设备等操作都需要通过系统调用进入内核完成。

4. C

解析：创建进程、终止进程、等待子进程、装入和执行程序等都属于进程控制类系统调用。

5. B

解析：单体内核通常把进程管理、内存管理、文件系统、设备驱动等核心功能放在同一个内核地址空间中执行，因此调用路径短、性能较高；但模块之间边界较弱，一个内核错误可能影响整个系统。

6. A

解析：微内核设计倾向于只在内核中保留最基础的机制，例如进程或线程管理、地址空间管理、进程间通信等。文件系统、网络协议栈、设备驱动等服务通常尽可能移到用户态。

7. B

解析：分层系统把操作系统划分为若干层，上层只依赖下层提供的服务。这样可以形成清晰的抽象和接口，便于理解、调试和维护。

8. B

解析：可加载内核模块可以在不重新编译整个内核的情况下动态加载或卸载功能，例如设备驱动或文件系统模块。它仍运行在内核态，因此保留较高执行效率，同时具有一定模块化能力。

9. A

解析：虚拟机的核心思想是通过虚拟化层把一套物理硬件抽象成多个虚拟硬件环境，使多个操作系统实例可以共享底层资源并相互隔离。

10. B

解析：系统启动时，引导程序会初始化基本硬件环境，找到并加载操作系统内核，然后把控制权交给内核。操作系统并不是开机后天然存在于 RAM 中。

## 二、简答题

### 1. 操作系统服务的两类划分

面向用户便利性的服务主要包括：

1. 用户界面：如命令行 Shell、图形界面、批处理接口，帮助用户与系统交互。
2. 程序执行：负责把程序装入内存、创建进程、开始执行并处理正常或异常终止。
3. I/O 操作：为程序提供统一的设备访问接口，使程序不必直接操作复杂硬件。
4. 文件系统操作：支持创建、删除、读写、重命名文件和目录。
5. 通信：支持同一主机或不同主机上的进程交换数据。
6. 错误检测：检测 CPU、内存、I/O 设备、程序运行中的错误并尽量处理。

面向系统高效运行的服务主要包括：

1. 资源分配：在多个进程或用户竞争 CPU、内存、I/O 设备等资源时进行分配。
2. 记账：记录用户或进程使用资源的情况，用于统计、计费或性能分析。
3. 保护与安全：防止进程越权访问资源，保护系统和用户数据。

二者的关注点不同：前一类更关注用户和应用程序是否方便、能否完成具体任务；后一类更关注系统整体是否公平、高效、可靠和安全。

### 2. 系统调用接口中 API、运行时库与内核的关系

系统调用接口是应用程序请求内核服务的边界。用户程序通常并不直接写底层系统调用指令，而是调用 API，例如 POSIX API、Win32 API 或 Java API。

大致关系如下：

1. 应用程序调用某个高级 API，例如 `read()`、`fork()`、`CreateFile()`。
2. API 通常由运行时库或系统库提供封装。运行时库负责检查参数、准备系统调用号和参数，并按照约定把参数放入寄存器、栈或内存块中。
3. 运行时库执行陷入指令，使 CPU 从用户态切换到内核态。
4. 内核中的 system-call interface 根据系统调用号找到对应的内核服务例程。
5. 内核完成实际操作后，把返回值或错误码交回运行时库，再由运行时库返回给应用程序。

因此，API 是程序员看到的较稳定、较高级的接口；system-call interface 是操作系统内核真正处理请求的入口集合；运行时库连接二者，屏蔽了陷入内核和参数传递的细节。

### 3. 分层操作系统结构的优点、局限性和教学价值

优点：

1. 模块化清晰：每一层只依赖下层服务，系统结构容易理解。
2. 便于调试：底层通过测试后，上层可以建立在相对可靠的基础之上。
3. 便于维护和扩展：接口明确时，某一层内部实现可以在较小范围内修改。
4. 有利于验证：层次边界清楚，更容易分析某个模块的正确性。

局限性：

1. 层次划分困难：很多操作系统功能天然存在交叉依赖，不容易严格归入某一层。
2. 性能可能下降：请求如果必须逐层穿越，会增加额外调用开销。
3. 灵活性较弱：严格分层可能限制上层直接使用某些底层能力。
4. 真实系统往往不完全分层：为了性能和工程需要，实际内核常采用混合结构。

在教学操作系统中采用分层设计很有价值。它能帮助学生从硬件抽象、内存管理、进程管理、文件系统到用户接口逐步理解操作系统，也能让实验按模块展开，降低一次性理解整个内核的难度。

## 三、分析题

### 1. 文件压缩服务的两种实现方案比较

方案 a 是把压缩功能直接编入内核；方案 b 是把压缩服务放在用户态，通过系统调用与内核交互。

从安全性看，内核态方案风险更高。压缩代码一旦出现越界访问、空指针、死循环等错误，可能导致整个系统崩溃或被攻击者利用获得内核权限。用户态方案的隔离性更好，服务崩溃通常只影响自身进程，操作系统可以通过权限、地址空间隔离和进程管理限制其危害。

从性能看，内核态方案可能更快。它可以减少用户态和内核态切换，减少数据在内核和用户空间之间的复制，适合与文件系统读写路径紧密结合的透明压缩。用户态方案需要通过系统调用读写文件或设备，可能有额外上下文切换和数据复制开销。不过压缩本身往往是 CPU 密集型任务，如果数据量不是极端大，这些额外开销未必是主要瓶颈。

从可维护性看，用户态方案更好。用户态程序可以更方便地调试、替换算法、升级版本，也可以复用已有压缩库。内核态代码开发门槛高、调试困难，对稳定性要求更严格，升级通常也更谨慎。

如果压缩只是普通文件工具或教学实验服务，我更倾向于选择用户态方案，因为它安全、易维护、便于迭代。如果需要实现文件系统级透明压缩、网络包路径上的实时压缩，或压缩功能必须和内核 I/O 路径高度结合且性能要求非常高，可以考虑把经过严格验证的核心部分放入内核。
