## 二、简答题

### 1. 机械磁盘访问时间的组成

寻道时间是磁臂把磁头移动到目标磁道所需要的时间。它反映的是机械移动开销，通常与当前磁头位置和目标磁道之间的距离有关。

旋转延迟是磁头到达目标磁道后，等待目标扇区随盘片旋转到磁头下方所需要的时间。平均旋转延迟通常约等于磁盘旋转一圈时间的一半。

传输时间是目标扇区到达磁头下方后，实际把数据从磁盘传到内存或从内存写到磁盘所需要的时间。它与数据量、磁盘转速、接口带宽等因素有关。

磁盘调度主要关注减少寻道距离，是因为寻道时间包含机械移动，通常比纯数据传输更慢、更不稳定。操作系统可以通过改变请求服务顺序来减少磁头在磁道之间来回移动的距离，从而提高吞吐量并降低平均响应时间。旋转延迟也会影响性能，但普通磁盘调度算法更容易直接控制的是磁道访问顺序，也就是寻道距离。

### 2. FCFS、SSTF、SCAN 与 C-SCAN 的比较

FCFS 按请求到达顺序服务。它实现简单、公平性好，不会因为位置远近而饿死某个请求；但它完全不考虑磁头当前位置，可能导致磁头在磁盘两端频繁往返，平均寻道距离较大。

SSTF 每次选择距离当前磁头最近的请求。它通常能显著降低平均寻道距离，提高吞吐量；但它偏向当前磁头附近的请求，远处请求可能长期等待，公平性较差，存在饥饿风险。

SCAN 沿一个方向移动磁头，服务沿途请求，到达端点后再反向移动。它比 FCFS 更能减少无序往返，也比 SSTF 更公平；但靠近两端的请求等待模式可能与中间磁道不同，响应时间仍不完全均匀。

C-SCAN 只在一个方向上服务请求，移动到端点后快速回到另一端，再继续同方向服务。它把磁道看成循环结构，可以让等待时间分布更均匀，适合追求公平性的场景；但回到起点的移动会增加额外磁头移动距离。

### 3. FCB/inode、目录项与路径名的关系

文件控制块 FCB 或 inode 是文件系统用于描述一个文件的核心元数据结构。它通常保存文件类型、权限、所有者、所属组、文件大小、创建/修改/访问时间、链接计数、数据块地址或索引块地址，以及一些状态标志等信息。

目录项保存的是名称到文件控制块的映射。在类 UNIX 文件系统中，目录项通常保存“文件名 -> inode 编号”；inode 中再保存文件元数据和定位文件数据块所需的信息。

路径名是用户看到的层次化文件定位方式，例如 `/home/user/a.txt`。文件系统解析路径名时，会从根目录或当前目录开始，逐级查找目录项：先在根目录中找到 `home` 对应的 inode，再在 `home` 目录中找到 `user`，最后在 `user` 目录中找到 `a.txt` 对应的 inode。最终，操作系统通过该 inode 找到文件元数据和数据块。

### 4. 三种文件分配方法的比较

连续分配把一个文件的所有块放在一段连续磁盘空间中。它的顺序访问和随机访问性能都很好，因为只要知道起始块号和偏移，就可以直接计算目标块地址；缺点是容易产生外部碎片，文件增长也不方便，可能需要移动文件或预留空间。

链式分配把文件块分散存放，每个块中保存指向下一个块的指针。它顺序访问较方便，不需要连续空间，因此没有外部碎片，文件增长也比较容易；但随机访问性能差，因为访问第 n 个块通常要从头沿链表走过去。此外，如果某个指针损坏，后续块可能丢失，可靠性较弱。

索引分配为文件建立索引块，索引块中保存文件各数据块的位置。它既不要求数据块连续，也支持较高效的随机访问；缺点是需要额外索引块空间，小文件可能有一定索引开销。如果索引块损坏，会影响对文件数据块的定位，因此需要缓存、备份或日志等机制提高可靠性。

## 三、分析题

### 1. 磁盘调度算法

磁道范围为 0 到 199，当前磁头在 53，请求队列为：

```text
98, 183, 37, 122, 14, 124, 65, 67
```

#### （a）（b）服务顺序与总磁头移动距离

FCFS 按到达顺序服务：

```text
53 -> 98 -> 183 -> 37 -> 122 -> 14 -> 124 -> 65 -> 67
```

总移动距离为：

```text
|98-53| + |183-98| + |37-183| + |122-37|
+ |14-122| + |124-14| + |65-124| + |67-65|
= 45 + 85 + 146 + 85 + 108 + 110 + 59 + 2
= 640
```

因此 FCFS 的服务顺序为：

```text
98, 183, 37, 122, 14, 124, 65, 67
```

总磁头移动距离为 640。

SSTF 每次选择距离当前磁头最近的请求：

```text
53 -> 65 -> 67 -> 37 -> 14 -> 98 -> 122 -> 124 -> 183
```

总移动距离为：

```text
12 + 2 + 30 + 23 + 84 + 24 + 2 + 59 = 236
```

因此 SSTF 的服务顺序为：

```text
65, 67, 37, 14, 98, 122, 124, 183
```

总磁头移动距离为 236。

SCAN 初始方向为磁道号增大方向，先向右服务所有较大磁道请求，到达端点 199 后反向：

```text
53 -> 65 -> 67 -> 98 -> 122 -> 124 -> 183 -> 199 -> 37 -> 14
```

服务请求的顺序为：

```text
65, 67, 98, 122, 124, 183, 37, 14
```

总移动距离为：

```text
(199 - 53) + (199 - 14) = 146 + 185 = 331
```

所以 SCAN 的总磁头移动距离为 331。

C-SCAN 初始方向同样为磁道号增大方向，只在增大方向服务请求。到达 199 后回到 0，再继续向增大方向服务剩余请求：

```text
53 -> 65 -> 67 -> 98 -> 122 -> 124 -> 183 -> 199 -> 0 -> 14 -> 37
```

服务请求的顺序为：

```text
65, 67, 98, 122, 124, 183, 14, 37
```

总移动距离为：

```text
(199 - 53) + (199 - 0) + (37 - 0)
= 146 + 199 + 37
= 382
```

所以 C-SCAN 的总磁头移动距离为 382。

汇总如下：

| 算法 | 服务顺序 | 总磁头移动距离 |
| --- | --- | --- |
| FCFS | 98, 183, 37, 122, 14, 124, 65, 67 | 640 |
| SSTF | 65, 67, 37, 14, 98, 122, 124, 183 | 236 |
| SCAN | 65, 67, 98, 122, 124, 183, 37, 14 | 331 |
| C-SCAN | 65, 67, 98, 122, 124, 183, 14, 37 | 382 |

#### （c）吞吐量与公平性的权衡

磁头移动距离越小，单位时间内通常能完成更多请求，吞吐量越高。在本题中，SSTF 的移动距离最小，因此从磁头移动开销看吞吐量最好。

但是，SSTF 过度偏向当前磁头附近的请求，远处请求可能长期等待，公平性较差。FCFS 严格按照到达顺序服务，公平性最好，但移动距离最大，吞吐量较差。SCAN 和 C-SCAN 处于二者之间：它们通过方向性扫描减少无序往返，同时让请求在较可预测的时间内得到服务。C-SCAN 的等待时间更均匀，但可能因为回到起点产生额外移动。

### 2. inode 索引结构

磁盘块大小为 4KB，每个块指针占 4B。

#### （a）

一个一级间接块中能保存的指针数为：

```text
4KB / 4B = 4096B / 4B = 1024
```

所以一个一级间接块最多可以索引 1024 个数据块。

#### （b）

直接块和一级间接块一共能表示的数据块数为：

```text
10 + 1024 = 1034
```

最大文件大小为：

```text
1034 * 4KB = 4136KB = 4,235,264B
```

所以该 inode 通过直接块和一级间接块最多可表示 4136KB 的文件。

#### （c）

文件大小为 50KB，需要的数据块数为：

```text
ceil(50KB / 4KB) = ceil(12.5) = 13
```

前 10 个数据块可以使用 10 个直接块指针，剩余 3 个数据块需要通过一级间接块指针索引。因此需要 13 个数据块，会使用直接块指针和一级间接块指针，不需要使用二级间接块指针。

#### （d）

50KB 文件共有 13 个数据块，最后 1KB 位于第 13 个数据块中。由于前 10 个块由直接指针表示，第 11 到第 13 个块需要通过一级间接块定位。

题目说明 inode 已在内存中，但索引块和数据块均未缓存。因此读取最后 1KB 至少需要：

1. 读取一级间接索引块，得到第 13 个数据块的块号。
2. 读取第 13 个数据块，再从中取出最后 1KB。

所以最少需要 2 次磁盘块读取。

### 3. 位图空闲空间管理

位图按块号 0 到 63 顺序为：

```text
11110000 11001100 00001111 11111100
00000000 10101010 11110000 00001111
```

其中 0 表示空闲，1 表示已分配。

#### （a）

从低块号开始扫描，前 10 个空闲块号为：

```text
4, 5, 6, 7, 10, 11, 14, 15, 16, 17
```

#### （b）

连续分配要求找到 6 个连续空闲块。从低块号开始扫描，块 14 到块 19 均为空闲：

```text
14, 15, 16, 17, 18, 19
```

因此可以为该文件分配 6 个连续块，起始块号为 14。

#### （c）

索引分配不要求数据块连续，只需要额外分配 1 个索引块来保存 6 个数据块的地址。一种可行方案是：

```text
索引块：4
数据块：5, 6, 7, 10, 11, 14
```

这些块在当前位图中均为空闲，因此该方案可行。

#### （d）

突然断电可能导致位图和目录/inode 信息不一致。例如，位图已经把某些块标记为已分配，但 inode 还没有记录这些块；或者 inode 已经引用了某些块，但位图仍显示为空闲。

日志文件系统通过把文件系统元数据更新组织成事务，先把即将进行的修改写入日志，再真正修改位图、inode 和目录等结构。系统崩溃后，可以根据日志重放已提交事务或撤销未完成事务，使文件系统恢复到一致状态。

fsck 则是在系统重启后检查整个文件系统的一致性。它可以扫描目录、inode、数据块引用和位图，发现空闲位图与实际引用不一致、重复分配、孤立 inode、丢失块等问题，并通过重建位图、释放未引用块、连接孤立文件等方式修复或降低数据不一致风险。
