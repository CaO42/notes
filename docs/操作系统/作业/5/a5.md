### 二

#### 1

临界区指每个进程中访问和操作共享数据或资源的那段代码片段 。  

临界区问题的三个基本要求 ：  

互斥：如果一个进程正在其临界区内执行，其他任何进程都不能进入它们的临界区 。  

前进：如果没有进程在临界区内执行，且有进程希望进入，那么只有不在剩余区执行的进程可以参与决定谁将下一个进入临界区，且这种选择不能被无限期推迟 。  

有界等待：从一个进程提出进入临界区的请求后，到该请求被批准前，其他进程被允许进入其临界区的次数必须存在一个上限，防止发生进程饥饿 。  

#### 2

不适合作为通用同步原语的原因：“关中断”在多处理器系统中只能禁用当前处理器的中断，无法阻止其他处理器继续执行并并发访问共享内存资源，因此无法在多个CPU之间提供真正的互斥 。如果向所有处理器发送关中断信号，会导致极大的消息传递延迟，严重降低系统性能 。  

与硬件原子指令相比的局限：将关中断的权限交给用户级进程非常危险，可能导致系统时钟丢失或被恶意进程霸占系统 。相比之下，硬件原子指令比如 test_and_set可以在硬件级别锁住内存总线或缓存，能在不影响系统中断响应的前提下实现多处理器层面真正的内存排他性 。  

#### 3

二值信号量与计数信号量比较：

二值信号量：其值只能取 0 或 1，行为类似于互斥锁，主要用于实现临界区的并发互斥控制 。  

计数信号量：其值可以在一个不受限制的整型域中变动，通常初始化为可用资源的数量，用于控制对具有多个实例的某种共享资源的访问。  

非原子性造成的问题：如果没有原子性保证，多个进程并发执行 wait(S) 或 signal(S) 操作时会引发竞争条件 。例如，两个进程同时读取值为 1 的信号量，双双判定可以进入，将信号量减为 0 并同时进入临界区，从而彻底破坏互斥性；或者同时执行 signal 操作时发生结果覆盖，导致信号量的更新丢失 。  

#### 4

读写器锁是一种允许多个读者同时读取共享数据，但在写者写入时必须独占资源的同步机制 。  

仍可能发生死锁 。 理由：死锁的发生与具体的锁类型无关，只要满足死锁的四个必要条件即可发生死锁 。例如，当程序中存在读写器锁 A 和 B 时：线程 1 持有锁 A 的读锁并阻塞等待锁 B 的写锁，而线程 2 持有锁 B 的读锁并阻塞等待锁 A 的写锁。此时构成了循环等待，同样会导致程序死锁 。  

#### 5

安全状态、不安全状态与死锁状态的关系：

- 安全状态：系统能按照某种顺序安全序列为每个进程分配资源直到顺利完成。系统处于安全状态时绝对不会发生死锁 。  
- 不安全状态：系统中不存在这样的安全序列。不安全状态是发生死锁的必要条件，但不代表已经死锁，因为进程可能提前释放资源 。  
- 死锁状态：系统中的进程相互等待，导致全部永远阻塞。死锁状态一定是不安全状态的子集 。  

死锁发生的四个必要条件 ：  

1. 互斥
2. 占有并等待
3. 不可抢占
4. 循环等待

### 三

#### 1

##### a

`flag[]`：这是一个布尔数组，用来表示进程是否有意愿或准备好进入临界区。`flag[i] = true` 表示当前进程 i 想要进入临界区 。 `turn`：这是一个整型变量，用来解决两个进程同时想要进入临界区时的冲突，它表示目前“轮到”哪个进程拥有进入临界区的优先权 。  

##### b

- 互斥：若 P0 和 P1 都在临界区，说明两者 `flag` 均为 true。但 `turn` 的值在任意时刻只能是 0 或 1，不可能同时满足两者的 while 循环持续条件，因此最多只能有一个进程突破循环进入临界区 。  
- 前进：如果 P1 不想进入（`flag[1] = false`），P0 的 while 条件直接不成立，P0 顺畅进入；如果双方都想进入（`flag` 均为 true），后执行 `turn = j` 的进程会把 `turn` 的值设为对方，导致先执行的进程突破循环，不会发生互相无限阻塞 。  
- 有界等待：假设 P0 正在等待，P1 在临界区。当 P1 退出临界区时，会将 `flag[1] = false`，从而允许 P0 进入临界区。即使 P1 瞬间再次尝试进入并将其 `flag` 设为 true，它也会主动将 `turn` 设为 0，这同样会打破 P0 的 while 循环让 P0 优先进入，因此进程的等待次数存在严格上限 。  

##### c

现代多处理器系统通常采用乱序执行机制或利用 Store Buffer 对内存写入进行优化 。在没有显式使用内存屏障的情况下，指令 `flag[i] = true` 和 `turn = j` 的写入在物理内存中的可见性顺序可能发生颠倒重排 。这可能导致两个核心同时读到对方的 `flag == false`，进而双双进入临界区，导致 Peterson 算法的互斥性失效 。  

#### 2

C

```c++
// 全局变量和信号量初始化
int north_count = 0;
int south_count = 0;
Semaphore north_mutex = 1; // 保护 north_count 计数器
Semaphore south_mutex = 1; // 保护 south_count 计数器
Semaphore bridge = 1;      // 桥梁整体使用权互斥锁

// 南行车辆
void vehicle_south() {
    wait(south_mutex);
    south_count++;
    if (south_count == 1) { 
        wait(bridge);  // 第一辆南行车负责锁住桥面
    }
    signal(south_mutex);

    // 车辆过桥

    wait(south_mutex);
    south_count--;
    if (south_count == 0) {
        signal(bridge); // 最后一辆南行车负责释放桥面
    }
    signal(south_mutex);
}

// 北行车辆
void vehicle_north() {
    wait(north_mutex);
    north_count++;
    if (north_count == 1) {
        wait(bridge);  // 第一辆北行车负责锁住桥面
    }
    signal(north_mutex);

    // 车辆过桥

    wait(north_mutex);
    north_count--;
    if (north_count == 0) {
        signal(bridge); // 最后一辆北行车负责释放桥面
    }
    signal(north_mutex);
}
```

不会出现相反方向车辆同时上桥的原因： 桥梁的整体使用权被 `bridge` 信号量严格互斥控制（初始值为1） 。当某方向（例如南向）第一辆车准备上桥时，必须执行 `wait(bridge)` 成功抢占桥梁资源。只要南向车辆未清空，`bridge` 的值就始终为 0 。此时如果北向有车尝试上桥，其第一辆车在执行 `wait(bridge)` 时必定会被阻塞，直到南向最后一辆车驶离并执行 `signal(bridge)` 。因此彻底排除了异向车辆同时在桥上的可能 。  

#### 3

##### a

根据 Need = Max - Allocation，各进程的 Need 矩阵计算如下 ： P0: 7 4 3 P1: 1 2 2 P2: 6 0 0 P3: 0 1 1 P4: 4 3 1  

##### b

当前系统处于安全状态 。 Available 为 (3, 3, 2) 。 安全序列推演如下：  

1. P1 Need (1, 2, 2) $\le$ Available (3, 3, 2)，执行 P1，释放后新 Available = (3, 3, 2) + (2, 0, 0) = (5, 3, 2)。
2. P3 Need (0, 1, 1) $\le$ Available (5, 3, 2)，执行 P3，释放后新 Available = (5, 3, 2) + (2, 1, 1) = (7, 4, 3)。
3. P4 Need (4, 3, 1) $\le$ Available (7, 4, 3)，执行 P4，释放后新 Available = (7, 4, 3) + (0, 0, 2) = (7, 4, 5)。
4. P0 Need (7, 4, 3) $\le$ Available (7, 4, 5)，执行 P0，释放后新 Available = (7, 4, 5) + (0, 1, 0) = (7, 5, 5)。
5. P2 Need (6, 0, 0) $\le$ Available (7, 5, 5)，执行 P2，全部分配完毕。 一个合法的安全序列为：<P1, P3, P4, P0, P2> 。  

##### c

系统能立即分配该请求 。 理由：  

1. Request1 (1, 0, 2) $\le$ Need P1 (1, 2, 2) 且 $\le$ 当前 Available (3, 3, 2)，请求合法且资源足够 。  
2. 进行试探性分配，更新状态：Available = (3, 3, 2) - (1, 0, 2) = (2, 3, 0)；P1 的 Allocation 变为 (3, 0, 2)，P1 的 Need 变为 (0, 2, 0) 。  
3. 检查系统安全性：当前 Available 为 (2, 3, 0)，满足 P1 的新 Need (0, 2, 0)。执行 P1 后释放其资源，Available = (2, 3, 0) + (3, 0, 2) = (5, 3, 2)。接着按顺序执行 P3 -> P4 -> P0 -> P2 依然可顺利完成，存在安全序列 <P1, P3, P4, P0, P2>，由于试探分配后系统依然处于安全状态，因此可以立即分配 。  

##### d

系统不能立即分配该请求 。 理由： 虽然 Request4 (3, 3, 0) $\le$ Need P4 (4, 3, 1) 且 $\le$ 当前 Available (3, 3, 2)，满足基础检查 。 但如果进行试探性分配，剩余 Available = (3, 3, 2) - (3, 3, 0) = (0, 0, 2) 。 此时，所有 5 个进程的 Need 中，A 类资源或 B 类资源的需求均大于 0，而当前 Available 中的 A 和 B 资源数量均为 0，导致没有任何一个进程能够获得足够的资源继续运行 。系统将陷入无法找到任何安全序列的不安全状态，因此该请求必须等待 。
