## 二、简答题

### 1. 临界区及临界区问题的三个基本要求

**临界区** 是进程或线程中访问共享数据、共享文件、共享设备或其他共享资源的代码段。多个执行流同时进入相关临界区，可能导致共享数据不一致，因此需要同步机制控制进入顺序。

临界区问题的正确解法通常需要满足三个要求：

1. **互斥**：如果一个进程正在临界区中执行，其他任何进程都不能进入与之相关的临界区。
2. **前进**：如果没有进程在临界区内执行，并且有进程希望进入临界区，那么系统必须能在有限时间内选择一个进程进入，不能无限期推迟。
3. **有界等待**：一个进程提出进入临界区的请求后，在它获得允许之前，其他进程进入临界区的次数必须有上界，防止饥饿。

### 2. 为什么关中断不适合作为多处理器系统中的通用同步原语？

在单处理器系统中，关中断可以阻止当前 CPU 被中断处理程序或调度器打断，因此能在很小范围内保护某些内核临界区。

但在多处理器系统中，它不适合作为通用同步原语，原因包括：

- 关闭一个 CPU 的中断，并不能阻止其他 CPU 上的进程或线程继续访问同一共享内存。
- 如果为了同步而关闭所有处理器的中断，代价很高，会严重影响系统响应性和并行性。
- 若允许用户程序关中断，会破坏操作系统对 CPU 的控制，甚至导致系统无法响应时钟和设备中断。

与硬件原子指令相比，关中断的局限更明显。`test_and_set()`、`compare_and_swap()` 等原子指令能在硬件层面对共享变量进行不可分割的读-改-写操作，适用于多处理器共享内存环境；而关中断主要只影响当前处理器的中断响应，不天然提供跨 CPU 的互斥。

### 3. 二值信号量与计数信号量，以及 wait/signal 原子性问题

**二值信号量** 的取值只有 0 和 1，常用于实现互斥锁。初值为 1 表示资源可用；进程进入临界区前执行 `wait(S)`，退出时执行 `signal(S)`。

**计数信号量** 的值可以是非负整数，通常表示某类资源的可用实例数。例如某资源有 5 个实例，则信号量可初始化为 5，每申请一个实例执行一次 `wait(S)`，释放一个实例执行一次 `signal(S)`。

`wait(S)` 和 `signal(S)` 必须原子执行。若不原子，可能出现以下问题：

- 两个进程同时看到信号量值为 1，并都认为自己可以进入临界区，导致互斥失效。
- 多个 `signal(S)` 操作同时发生时，更新结果可能相互覆盖，造成释放次数丢失。
- 信号量值与真实资源状态不一致，进而引起死锁、资源泄漏或并发错误。

### 4. 读写器锁及多个读写器锁下是否可能死锁

读写器锁是一种适合“读多写少”场景的同步机制：

- 多个读者可以同时持有读锁并读取共享数据。
- 写者必须独占写锁。
- 写者写入时，其他读者和写者都不能进入相应临界区。

即使程序只使用读写器锁，也仍可能发生死锁。死锁并不取决于锁的名字，而取决于是否满足死锁必要条件。

例如，系统中有两个读写器锁 A 和 B：

- 线程 T1 持有 A，等待 B。
- 线程 T2 持有 B，等待 A。

如果两个线程都不释放已持有的锁，就形成循环等待，从而发生死锁。读写器锁还可能因为读锁升级为写锁、多个锁获取顺序不一致等情况产生死锁。因此，使用多个读写器锁时仍需要统一加锁顺序，或使用超时、尝试加锁、锁层级等策略降低风险。

### 5. 安全状态、不安全状态和死锁状态的关系，以及死锁四个必要条件

**安全状态** 指系统存在至少一个安全序列。按照该序列，系统可以依次满足每个进程的最大资源需求，使它们完成并释放资源。处于安全状态说明系统可以避免死锁。

**不安全状态** 指系统找不到安全序列。不安全状态不等于已经死锁，但说明系统有可能在后续资源请求中进入死锁。

**死锁状态** 指一组进程互相等待对方持有的资源，导致这组进程都无法继续执行。死锁状态一定是不安全状态，但不安全状态不一定已经死锁。

关系可以概括为：

```text
安全状态：一定不是死锁，并且存在避免死锁的完成顺序。
不安全状态：可能导致死锁，但未必已经死锁。
死锁状态：一定属于不安全状态。
```

死锁发生的四个必要条件：

1. **互斥**：至少有一种资源一次只能被一个进程使用。
2. **占有并等待**：进程已经持有至少一个资源，同时还在等待其他资源。
3. **不可抢占**：资源不能被强制夺走，只能由持有者主动释放。
4. **循环等待**：存在一组进程 P0、P1、...、Pn，使 P0 等待 P1 持有的资源，P1 等待 P2 持有的资源，最终 Pn 又等待 P0 持有的资源。

## 三、分析题

### 1. Peterson 算法分析

代码如下：

```c
do {
    flag[i] = true;
    turn = j;
    while (flag[j] && turn == j)
        ;    /* busy wait */

    /* critical section */

    flag[i] = false;

    /* remainder section */
} while (true);
```

#### a. flag[] 和 turn 的作用

`flag[]` 用来表示进程是否想进入临界区。

- `flag[i] = true` 表示进程 i 想进入临界区。
- `flag[i] = false` 表示进程 i 不想进入临界区或已经离开临界区。

`turn` 用来在两个进程同时想进入临界区时打破冲突。

- 进程 i 在进入前执行 `turn = j`，表示如果双方都想进入，则把优先权让给对方。
- 当两个进程同时竞争时，最后写入 `turn` 的进程会等待，另一个进程先进入。

#### b. 顺序一致内存模型下如何满足三个要求

**互斥**：

假设 P0 和 P1 都想进入临界区，则 `flag[0]` 和 `flag[1]` 都为 `true`。此时 `turn` 只能等于 0 或 1。

- 若 `turn == 0`，P1 的等待条件 `flag[0] && turn == 0` 为真，P1 等待，P0 可进入。
- 若 `turn == 1`，P0 的等待条件 `flag[1] && turn == 1` 为真，P0 等待，P1 可进入。

因此两者不可能同时通过 while 循环进入临界区。

**前进**：

如果只有一个进程想进入临界区，例如 P0 想进入而 P1 不想进入，则 `flag[1] == false`，P0 的 while 条件为假，可以立即进入。

如果两个进程都想进入，`turn` 会决定其中一个等待，另一个进入，不会出现两个进程都无限等待而无人进入的情况。

**有界等待**：

当 P0 正在等待时，说明 P1 可能正在临界区或将要进入临界区。P1 退出临界区后会执行 `flag[1] = false`，使 P0 的等待条件变假。即使 P1 立刻再次请求进入，它也会执行 `turn = 0`，把优先权让给 P0。因此 P0 不会被 P1 无限次插队，等待次数有界。

#### c. 为什么现代多处理器还要考虑内存重排序或内存屏障？

Peterson 算法的正确性依赖顺序一致内存模型，即每个处理器看到的内存读写顺序与程序顺序一致。

现代多处理器系统为了提高性能，可能进行：

- 编译器指令重排。
- CPU 乱序执行。
- store buffer 导致写入对其他处理器延迟可见。
- 缓存一致性带来的可见性延迟。

如果 `flag[i] = true` 和 `turn = j` 的可见顺序被改变，两个处理器可能都没有及时看到对方的 `flag`，从而都认为可以进入临界区，破坏互斥。因此在现代系统中，实现这类算法通常需要内存屏障或使用语言和硬件提供的原子操作及内存序语义。

### 2. 单线桥信号量方案

要求：任意时刻桥上可以有多个同方向车辆，但不能同时存在南行和北行车辆。无需解决饥饿问题。

可以把同方向车辆看作同类读者：同方向第一辆车负责锁住桥，最后一辆车负责释放桥。

共享变量和信号量：

```c
int north_count = 0;
int south_count = 0;

semaphore north_mutex = 1;  // 保护 north_count
semaphore south_mutex = 1;  // 保护 south_count
semaphore bridge = 1;       // 桥的方向互斥权
```

北行车辆：

```c
void north_vehicle() {
    wait(north_mutex);
    north_count++;
    if (north_count == 1) {
        wait(bridge);       // 第一辆北行车锁住桥
    }
    signal(north_mutex);

    // cross bridge northbound

    wait(north_mutex);
    north_count--;
    if (north_count == 0) {
        signal(bridge);     // 最后一辆北行车释放桥
    }
    signal(north_mutex);
}
```

南行车辆：

```c
void south_vehicle() {
    wait(south_mutex);
    south_count++;
    if (south_count == 1) {
        wait(bridge);       // 第一辆南行车锁住桥
    }
    signal(south_mutex);

    // cross bridge southbound

    wait(south_mutex);
    south_count--;
    if (south_count == 0) {
        signal(bridge);     // 最后一辆南行车释放桥
    }
    signal(south_mutex);
}
```

为什么不会出现相反方向车辆同时上桥：

- `bridge` 初值为 1，同一时刻只能被一个方向的第一辆车成功获得。
- 某方向第一辆车获得 `bridge` 后，该方向后续车辆只需更新本方向计数即可继续上桥。
- 在该方向车辆数量降为 0 之前，`bridge` 不会释放。
- 相反方向的第一辆车必须等待 `bridge`，因此不能与当前方向车辆同时上桥。

该方案不保证公平性。如果某一方向车辆持续到达，另一方向可能长期等待；若要解决饥饿，需要额外加入排队、公平信号量或方向切换策略。

### 3. 银行家算法

已知：

| 进程 | Allocation (A B C) | Max (A B C) |
| :--- | :----------------- | :---------- |
| P0 | 0 1 0 | 7 5 3 |
| P1 | 2 0 0 | 3 2 2 |
| P2 | 3 0 2 | 9 0 2 |
| P3 | 2 1 1 | 2 2 2 |
| P4 | 0 0 2 | 4 3 3 |

```text
Available = (3, 3, 2)
Need = Max - Allocation
```

#### a. Need 矩阵

| 进程 | Need (A B C) |
| :--- | :----------- |
| P0 | 7 4 3 |
| P1 | 1 2 2 |
| P2 | 6 0 0 |
| P3 | 0 1 1 |
| P4 | 4 3 1 |

#### b. 判断当前系统是否安全

初始：

```text
Work = Available = (3, 3, 2)
```

寻找安全序列：

1. P1 的 Need = (1, 2, 2) <= Work = (3, 3, 2)，可完成。释放 Allocation(P1) = (2, 0, 0)，Work 变为 (5, 3, 2)。
2. P3 的 Need = (0, 1, 1) <= Work = (5, 3, 2)，可完成。释放 Allocation(P3) = (2, 1, 1)，Work 变为 (7, 4, 3)。
3. P4 的 Need = (4, 3, 1) <= Work = (7, 4, 3)，可完成。释放 Allocation(P4) = (0, 0, 2)，Work 变为 (7, 4, 5)。
4. P0 的 Need = (7, 4, 3) <= Work = (7, 4, 5)，可完成。释放 Allocation(P0) = (0, 1, 0)，Work 变为 (7, 5, 5)。
5. P2 的 Need = (6, 0, 0) <= Work = (7, 5, 5)，可完成。

因此当前系统处于安全状态，一个安全序列为：

```text
<P1, P3, P4, P0, P2>
```

#### c. P1 请求 Request1 = (1, 0, 2)，能否立即分配？

先做合法性检查：

```text
Request1 = (1, 0, 2) <= Need(P1) = (1, 2, 2)
Request1 = (1, 0, 2) <= Available = (3, 3, 2)
```

请求没有超过 P1 最大需求，也没有超过当前可用资源，可以进行试探性分配。

试探分配后：

```text
Available = (3, 3, 2) - (1, 0, 2) = (2, 3, 0)
Allocation(P1) = (2, 0, 0) + (1, 0, 2) = (3, 0, 2)
Need(P1) = (1, 2, 2) - (1, 0, 2) = (0, 2, 0)
```

安全性检查：

1. Work = (2, 3, 0)，P1 Need = (0, 2, 0) <= Work，P1 可完成并释放 (3, 0, 2)，Work = (5, 3, 2)。
2. P3 Need = (0, 1, 1) <= Work，完成后 Work = (7, 4, 3)。
3. P4 Need = (4, 3, 1) <= Work，完成后 Work = (7, 4, 5)。
4. P0 Need = (7, 4, 3) <= Work，完成后 Work = (7, 5, 5)。
5. P2 Need = (6, 0, 0) <= Work，完成。

试探分配后仍存在安全序列：

```text
<P1, P3, P4, P0, P2>
```

所以 P1 的请求可以立即分配。

#### d. P4 请求 Request4 = (3, 3, 0)，能否立即分配？

先做合法性检查：

```text
Request4 = (3, 3, 0) <= Need(P4) = (4, 3, 1)
Request4 = (3, 3, 0) <= Available = (3, 3, 2)
```

请求没有超过最大需求，也没有超过当前可用资源，可以进行试探性分配。

试探分配后：

```text
Available = (3, 3, 2) - (3, 3, 0) = (0, 0, 2)
Allocation(P4) = (0, 0, 2) + (3, 3, 0) = (3, 3, 2)
Need(P4) = (4, 3, 1) - (3, 3, 0) = (1, 0, 1)
```

此时 Work = (0, 0, 2)。检查各进程 Need：

| 进程 | Need (A B C) | 是否 <= Work (0 0 2) |
| :--- | :----------- | :------------------- |
| P0 | 7 4 3 | 否 |
| P1 | 1 2 2 | 否 |
| P2 | 6 0 0 | 否 |
| P3 | 0 1 1 | 否 |
| P4 | 1 0 1 | 否 |

没有任何进程能够在当前 Work 下完成，因此找不到安全序列。试探分配会使系统进入不安全状态。

所以 P4 的请求不能立即分配，必须等待。
